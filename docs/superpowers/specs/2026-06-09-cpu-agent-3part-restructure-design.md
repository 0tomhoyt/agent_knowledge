## 1. 重构动机与统一主线

### 1.1 为什么重构

旧结构是"问题驱动"（瓶颈分析 → 前瞻方向），四部分、27 正文页，三个独立 mini-talk 之间叙事弧偏散。作者希望改为**三部分**：

1. **Agent 执行流程的基础分享介绍**（主打基础）
2. **Agent-to-Agent 洞察报告**
3. **通信计算融合在 ARM CPU、SME、SVE 下的潜在机会**

并在三个关键约束下重构：

- **硬件**：团队主力平台为 **华为鲲鹏 920（TaiShan v110, ARMv8.2-A；现役仅 NEON 128-bit，无 SVE/SVE2/SME）**（✅ 多源确认：IEEE Micro 2021 + VLDB Journal 2022 + Stony Brook Ookami）；同时团队已通过 early-access 取得**新一代鲲鹏（内部可得，⚠️ 暂无公开来源）= full SVE + SVE2 + SME**。**R1 决议口径（2026-06-10 再决议）= 鲲鹏 920 NEON-only（✅）+ 新一代鲲鹏 SVE/SVE2/SME（⚠️ early-access 已可得）**（见 §11、`docs/superpowers/research/2026-06-09-platform-sme-sve.md`）。第三部分因此是"NEON（鲲鹏 920 现役）+ SVE/SVE2/SME（新一代鲲鹏 early-access）从今天起即可在自有 SVE/SME 硅上实测，无需再等未来平台"。
- **P2 定位**："洞察报告" = **两者结合**——先快速盘点业界/学术现状（建立共同认知），再落到团队提炼的关键洞察与瓶颈，桥到第三部分。
- **时间**：**均衡分配**，约 P1 8min / P2 10min / P3 9min + 开场 1min + 总结 2min = 30min。

### 1.2 统一主线（全场一句话 spine）

> **Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 从单 Agent 的"工具调用往返"，到多 Agent 的"A2A 协作"，**通信占比随规模上升、逐渐压过计算成为主导成本**。而我们的 ARM 平台——**NEON（鲲鹏 920 现役）+ SVE/SVE2/SME（新一代鲲鹏 early-access 现已可得）是同一融合方向——武器已齐备，团队可从今天起在自有 SVE/SME 硅上实测验证，无需再等未来平台**——它不只是算子加速器，更是能把"通信的数据搬运/序列化"与"计算的向量化/矩阵化"放进**同一组执行单元**的融合点。**通信计算融合**，就是我们在 Agent 时代的主场。

三部分构成 **problem setup → diagnosis → prescription** 的诊断弧：

| 部分 | 在主线中的角色 |
|------|----------------|
| **P1 基础** | **problem setup** — 让听众从第 1 页就看到 agent loop 本质是 compute↔comm 交织（非"什么是大模型"那种浅基础） |
| **P2 洞察** | **diagnosis** — 现状盘点 + 数据论证："单→多 Agent，通信占比上升、渐成主导" |
| **P3 融合** | **prescription** — 在 ARM 平台（NEON 现役·鲲鹏 920 + SVE/SVE2/SME early-access·新一代鲲鹏）把 comm 与 compute 融合 = 主场机会 |

主线把作者最初对 P3 的措辞"潜在机会"升级为"有论证的主张"——机会不是清单，而是诊断弧的自然结论。

## 3. Part 1 基础 —— Agent 执行流程基础（5 页, 8min, problem setup）

**目标**：结束时听众把 agent loop 看成"计算↔通信的反复交织"，并理解为什么这是新 workload。**不讲"什么是大模型/神经网络"**——受众是系统软件工程师，要的是 agent 运行时视角。

| 页 | 标题 | 要点 | 主线触点 | 来源 |
|----|------|------|----------|------|
| P1-1 | 章节封面 + 划边界 | 一句话定义"Agent 运行时"；我们不碰训练/推理内核，讲 loop 里被 LLM 光环掩盖的系统层 | — | 新写 |
| P1-2 | Agent 执行流程 = 一个循环 | 感知→推理→规划→工具调用→记忆更新 循环图；不是"一次 API"而是几十~几百步长循环 | — | 承袭（SWE-bench 250步/$3 上限） |
| P1-3 | 拆开循环：每步是什么计算形态 | 感知(文本处理)/推理(LLM)/规划/工具调用(重计算·IO)/记忆(向量检索·rerank)→映射到 CPU/内存/通信需求表 | — | 承袭 |
| **P1-4** | **loop = 计算↔通信的反复交织** | 把循环重画成"本地计算 ↔ 跨边界通信(API/IPC/工具)"的乒乓；API 等待 500ms-5s 间隙本地 CPU 利用率~0% | **🎯 主线第一次点题** | 承袭（标注数量级） |
| P1-5 | 为什么是"新"workload | 混合/突发/长尾/依赖复杂/fail-slow（成功 12 步/$1.21 vs 全部 21 步/$2.52）；传统调度假设失效 | 为 P2 diagnosis 铺路 | 承袭（SWE-agent） |

**承袭率**：P1 几乎全是现有已验证内容的**蒸馏**（SWE-bench、SWE-agent、API 间隙 CPU~0%、workload 特征均已过校验），新研究量很低。P1-4 是主线 spine 的第一次显式亮相。

**可选第 6 页**（若 5 页讲 8min 偏松）：一张"本地 CPU 在忙什么"profiling 示意图（串行流水线灰色空闲 vs 并行填满），视觉化优化目标。

## 4. Part 2 洞察 —— Agent-to-Agent 洞察报告（7 页, 10min, diagnosis）

**角色**：主线的 **diagnosis**。按"两者结合"——**先现状盘点（建立共同认知）→ 再我们提炼的洞察 → 桥到 P3**。诊断论断：**"从单 Agent 到多 Agent，通信占比随规模上升、渐成主导。"**

| 页 | 标题 | 要点 | 主线触点 | 来源 |
|----|------|------|----------|------|
| P2-1 | 章节封面 + 诊断问题 | "从 1 个 agent 到 N 个 agent，成本结构怎么变？"——引出"通信"这个新变量 | — | 新写 |
| P2-2 | 现状盘点：A2A 正在发生什么 | 主流多 Agent 框架速览（AutoGen/MetaGPT/CrewAI/LangGraph）+ Google **A2A 协议**(2025) + 学术 multi-agent 基准；一张"地图"建立共同认知 | — | **🆕 需研究** |
| P2-3 | 关键数据：多 Agent 比单 Agent 贵多少 | token/成本 ~2-5×（经验法则，**标注非精确**）；调度气泡可达 ~58%（Continuum, arXiv:2511.02230）；最慢 agent 决定端到端延迟 | 数据铺路 | 承袭（**需重标口径**） |
| P2-4 | 我们的洞察①：A2A 数据怎么流 | 点对点/广播/层级编排/共享黑板；消息序列化开销（JSON vs Protobuf vs FlatBuffers）；随 agent 数通信复杂度 O(N²) vs O(N log N) | 通信成本在哪 | 承袭 |
| P2-5 | 我们的洞察②：被锁与不均吃掉的延迟 | 锁竞争 10-100× 退化（**已确认✅**）；分布式共识延迟（Raft/Paxos ms 级）；负载不均衡（最慢 agent 拖累整体） | 通信成本在哪 | 承袭 |
| **P2-6** | **诊断合成：通信随规模上升、渐成主导** | 一张图：单 agent（通信占比小）→ 小 swarm → 大 swarm（通信占比主导）；把 P2-3/4/5 收成一句话论断 | **🎯 diagnosis 落地** | 综合 |
| **P2-7** | **桥到 P3：通信变贵了，谁来付？** | 只抛问题、不回答："如果通信是上升中的主导成本，而我们 ARM 平台能把通信和计算放进同一组 SME/SVE 单元……" | **🎯 主线第二次点题（handoff）** | 新写 |

**承袭与新研究**：
- **承袭为主**：P2-4（通信模式/序列化）、P2-5（锁/共识/负载）几乎全是现有已验证内容。
- **P2-3 必须重标口径**：仓库校验报告已警告——"2-5×"是 token/成本经验法则、"58%"是调度气泡而非工具调用时间。PPT 按**数量级 + 限定语境**引用，不能当精确测量。
- **🆕 新研究集中一处**：P2-2 的 A2A 现状盘点（框架生态 + A2A 协议 + 学术基准）是现有文档没有的。

**叙事设计**：P2-7 是"只问不答"的 handoff 页——制造对 P3 的期待，把诊断张力推到最高，由 P3 给出 prescription。
