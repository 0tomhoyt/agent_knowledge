## 速查：各领域的 load-bearing insight
- **P2-2**: A2A/multi-agent 生态系统分为三层——框架通信拓扑差异（AutoGen 广播/MetaGPT 全局消息池/CrewAI 层级委托/LangGraph 状态图）、Google A2A 协议标准化跨厂商互操作（HTTP + JSON-RPC 2.0 + SSE）、学术基准量化多 agent 真实成本（UIUC: prefill tokens 4-220x，Continuum: scheduling bubble + KV-cache TTL）。CPU/ARM 团队的核心洞察：A2A 是纯 Web 传输（CPU 处理 TLS/JSON，无特殊硬件需求），性能痛点在服务层 prefill 膨胀和缓存重调度气泡，而非 agent 逻辑本身。
- **P3-3**: 计算-通信重叠是经典 HPC 延迟隐藏技术（MPI 非阻塞 Isend/Irecv），现代加速器训练仍依赖（FSDP/pipeline 并行），但重叠非免费——Georgia Tech 2025 实测显示平均 ~18.9% 计算减速（最高 40%）。序列化/marshaling 本身是可向量化计算（simdjson 多 GB/s，Open MPI 论文用 ARM SVE 加速 MPI datatype pack/unpack 达 ~2-4x）。ARM SVE gather/scatter 是不规则数据移动的指令级原语（2x 加速 pack/unpack，但 lattice QCD 显示高度分散模式"rather slow"）。节点本地 agent IPC 的极端端点是零拷贝共享内存（iceoryx2 亚微秒级）。Kunpeng 相关综合：必须移动的数据用 SVE/NEON 向量化 marshaling 层，不应移动的数据优先零拷贝共享内存 + DMA 优先级重叠。
- **P3-4**: Agent 运行时产生的小/瘦矩阵工作负载（单 agent 或小群体 embedding 查找、rerank 评分、每步 attention）在结构上不同于主流硬件和库优化的大型方阵批量 GEMM。ARMv8（当前 Kunpeng 920，仅 NEON 128-bit）上，LibXSMM 和 LibShaLom 通过 JIT 代码生成和降低 packing 开销专门 targeting 小/不规则 GEMM，因为传统 BLAS 在这些形状上表现不佳（packing 开销和向量通道利用率不足主导）。ARMv9 SME 路线图增加 2D ZA tile 和外积（MOPA）指令，将每核 FP32 吞吐量提升 ~4x（0.512 vs 0.128 TFLOPS）；但 DATE 2026 KirbyMM 论文（直接在 Kunpeng LX2 核上 benchmark，对抗 Kunpeng 自家 KBLAS）显示 SME 使大型稠密 GEMM 成为内存受限，小/边缘情况矩阵需要专用 SME+SVE 混合例程以避免浪费矩阵单元。MpGEMM 和 FlashDecoding++ 文档说明为何小批量/瘦推理低效：算术强度崩溃（decode 事实上是 GEMV，内存受限），扁平/瘦形状强制零填充浪费 ~50% 计算。Kunpeng 团队的可操作结论：NEON 时代的胜利来自专用小-GEMM kernel（LibXSMM/LibShaLom 风格）和批处理/swarm 放大；SME 时代的胜利需要混合策略——对计算受限块使用矩阵单元，但对瘦/边缘情况回退到 SVE/NEON，因为当 M 或 N 极小时矩阵单元无法摊销。
- **P3-5**: Agent-to-agent 和工具调用流量绝大多数是文本/JSON 形状，因此序列化和验证数据路径是现实的瓶颈，远早于 LLM 推理成本。simdjson 证明了 SIMD 向量化结构检测（阶段 1）+ 轻分支解码可让单个商用核以多 GB/s 解析完全校验的 JSON，并随 NEON 路径一起提供。在现代 ARM 服务器核上 NEON 路径已达到多 GB/s 量级（如 Daniel Lemire 测量：AWS Graviton 4/Neoverse V2 单核 ~4.6 GB/s），但 128-bit NEON 落后于更宽的 x86 向量且缺乏 gather/scatter。SVE/SVE2 路线图是缩小该差距的主要杠杆：可变向量长度（128-2048 bits，VLA）、谓词寄存器、原生 gather/scatter，以及专用字符串匹配指令如 SVMATCH（Bytedance Sonic JSON 解码器在 Neoverse-N2 上获得高达 ~24% 吞吐量提升）。对于非 JSON 路径，零拷贝格式（FlatBuffers/Cap'n Proto）通过以更大的线尺寸支付成本，将解码成本基本坍缩为指针解引用。对于 marshaling/layout 工作，canonical SIMD 变换是通过 NEON zip/unzip/transpose intrinsics 进行 AoS<->SoA 转置；在 NEON 上缺乏 gather/scatter 强制这种寄存器级重排，而 SVE/SVE2 移除了对非连续消息字段的该约束。

## P2-2: A2A 协议与 Multi-Agent 框架拓扑差异（通信层、互操作标准、性能成本）

A2A/multi-agent 生态系统分为三层。（1）框架通信拓扑差异：Microsoft AutoGen 使用广播/共享 topic 的 GroupChat 模型，由 GroupChatManager 协调；MetaGPT（ICLR 2024）在全局发布-订阅消息池上编码标准操作程序（SOPs）；CrewAI 使用基于角色的层级任务委托；LangGraph 将 agent 建模为有状态图（节点/边）而非自由对话。（2）Google 的 Agent2Agent（A2A）协议于 2025 年 4 月宣布，拥有 50+ 合作伙伴，标准化跨厂商互操作（纯 HTTP + JSON-RPC 2.0 + SSE），配备 Agent Cards（能力发现）和有状态 Task 生命周期（已提交/工作中/已完成），补充 Anthropic 的 MCP。（3）学术基准现在量化真实成本：UIUC 研究（arXiv:2505.18286）测量多 agent 系统在 7 个数据集/6 个模型上消耗单 agent 等效物的 4x-220x 更多 prefill token；Continuum（arXiv:2511.02230，ICLR 2026）形式化"scheduling bubble"——工具调用完成但 KV-cache 必须重新驻留时的空闲间隙——引入 KV-cache TTL 机制以削减多轮 agent 作业完成时间。对于 CPU/ARM（Kunpeng）团队，load-bearing洞察是 A2A 是纯 Web 传输（CPU 处理 TLS/JSON，无特殊硬件），而性能痛点在服务层 prefill 膨胀和缓存重调度气泡，而非 agent 逻辑本身。

## P3-3: 融合 Agent 运行时通信与计算（Kunpeng ARM CPU 上的计算-通信重叠、数据编组、SVE 原语、零拷贝 IPC）

关于在 Kunpeng（ARM）CPU 上融合 agent 运行时通信与计算的演讲，先验技术分为四个主题。（1）计算-通信重叠作为延迟隐藏技术是成熟的 HPC 思想（Sandia/Bernholdt，MPI 非阻塞 Isend/Irecv），并仍然是现代加速器训练中的主导策略（FSDP，pipeline 并行，AWS Neuron 的专用 CC 核心）；最近的 Georgia Tech 表征（arXiv 2507.03114）量化重叠仍有帮助（顺序比重叠慢 ~10%），但由于共享资源（DMA/内存带宽/功耗）争用导致真实计算减速（平均 ~19%，高达 40%）。（2）"数据编组 = 计算"思想——即序列化/pack-unpack/memcpy 本身是可向量化的计算——由 simdjson（多 GB/s SIMD JSON 解析，有 NEON/SVE 后端）和一篇 Open MPI 论文（Zhong/Dongarra 等，IEEE CCGrid 2020）直接证明，后者使用 ARM SVE 内联函数加速 MPI datatype pack/unpack 和归约，将消息运行时的数据移动视为向量计算 kernel。（3）ARM SVE gather-load / scatter-store 是不规则/非连续消息负载（稀疏字段、struct-of-arrays、派生 MPI datatypes）的指令级原语，在 A64FX 硬件上提供 2x pack/unpack 加速，指令计数减少 16% 到 ~10x（取决于向量长度）——尽管 gather/scatter 在高度分散模式上"方便但相当慢"（lattice QCD 警告）。（4）对于节点本地 agent IPC，真正的零拷贝共享内存（iceoryx2）通过完全消除编组达到亚微秒 / ~100ns 级延迟——这是融合轴的极端端点。Kunpeng 相关综合：必须移动的数据用 SVE/NEON 向量化编组/序列化层，不应移动的数据优先零拷贝共享内存 + DMA 优先级重叠。

## P3-4: Agent Runtime 小/瘦矩阵 GEMM（ARMv8/v9 SME 上的小矩阵乘法专用库、KirbyMM/Kunpeng 基准、MpGEMM/FlashDecoding++ 低效性洞察）

Agent 运行时产生小/瘦矩阵工作负载——单 agent 或小群体 embedding 查找、rerank 评分和每步 attention——在结构上不同于主流硬件和库优化的大型、方阵、批量放大的 GEMM。在 ARMv8（Kunpeng 920，当前仅 NEON 128-bit）上，LibXSMM 和 LibShaLom 等专用库通过 JIT 代码生成和降低 packing 开销明确针对小/不规则 GEMM，因为传统 BLAS 在这些形状上表现不佳（packing 开销和向量通道利用率不足主导）。路线图向 ARMv9 SME 的移动增加 2D ZA tile 和外积（MOPA）指令，将每核 FP32 吞吐量提升 ~4x；DATE 2026 KirbyMM 论文——直接在 Kunpeng 的 LX2 核上 benchmark，对抗 Kunpeng 自家 KBLAS——显示 SME 使大型稠密 GEMM 内存受限，小/边缘情况矩阵需要专用 SME+SVE 混合例程以避免浪费矩阵单元。关键地，MpGEMM（"Demystifying ARM SME" arXiv 2025 工作）和 FlashDecoding++（MLSys 2024）文档说明为何小批量/瘦推理低效：算术强度崩溃（decode 事实上是 GEMV，内存受限），扁平/瘦形状强制零填充浪费 ~50% 计算。Kunpeng 团队的可操作结论：NEON 时代的胜利来自专用小-GEMM kernel（LibXSMM/LibShaLom 风格）和批处理/swarm 放大；SME 时代的胜利需要混合策略——对计算受限块使用矩阵单元，但对瘦/边缘情况回退到 SVE/NEON，因为当 M 或 N 极小时矩阵单元无法摊销。

## P3-5: Agent 通信序列化/验证数据路径（ARM NEON/SVE 上 JSON/零拷贝/SIMD AoS<->SoA）

Agent-to-agent 和工具调用流量绝大多数是文本/JSON 形状，因此序列化和验证数据路径是现实的瓶颈，远早于 LLM 推理成本。simdjson 证明了 SIMD 向量化结构检测（阶段 1）+ 轻分支解码可让单个商用核以多 GB/s 解析完全校验的 JSON，并随 NEON 路径一起提供。在现代 ARM 服务器核上 NEON 路径已达到多 GB/s 量级（例如 Daniel Lemire 测量：AWS Graviton 4 / Neoverse V2 单核 ~4.6 GB/s），但 128-bit NEON 落后于更宽的 x86 向量且缺乏 gather/scatter。SVE/SVE2 路线图是缩小该差距的主要杠杆：可变向量长度（128-2048 bits，VLA）、谓词寄存器、原生 gather/scatter，以及专用字符串匹配指令如 SVMATCH（Bytedance Sonic JSON 解码器在 Neoverse-N2 上获得高达 ~24% 吞吐量提升）。对于非 JSON 路径，零拷贝格式（FlatBuffers/Cap'n Proto）通过以更大的线尺寸支付成本，将解码成本基本坍缩为指针解引用。对于 marshaling/layout 工作，canonical SIMD 变换是通过 NEON zip/unzip/transpose intrinsics 进行 AoS<->SoA 转置；在 NEON 上缺乏 gather/scatter 强制这种寄存器级重排，而 SVE/SVE2 移除了对非连续消息字段的该约束。
