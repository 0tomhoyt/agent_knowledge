========== PAGE 1: 开场：主线一句话 + 议程 ==========

## 核心观点（一句话）

**Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 从单 Agent 的"工具调用往返"，到多 Agent 的"A2A 协作"，**通信占比随规模上升、逐渐压过计算成为主导成本**。而我们的 ARM 平台——**现役 NEON 已能融合，SVE/SVE2/SME（鲲鹏 930+/950+ 解锁）是同一融合方向的下一站**——它不只是算子加速器，更是能把"通信的数据搬运/序列化"与"计算的向量化/矩阵化"放进**同一组执行单元**的融合点。**通信计算融合**，就是我们在 Agent 时代的主场。

## 详细要点

### 要点1：Agent 不是"更大的模型"，而是"带工具的长循环"

传统 LLM 调用是一次 request、一次 response。Agent 是把这个调用放进一个可能长达几十步、上百步的循环里。权威数据：SWE-bench Verified 的标准预算上限是每实例 **最多 250 步、最多 3 美元** [来源: Can Software Engineering Agents Self-Evolve on the Fly? arXiv 2511.13646, 引用 SWE-bench Verified 官方配置]；SWE-bench Pro (Scale AI) 给每个 agent 任务 **最多 50 次工具调用**，作者称该额度 "相当标准" [来源: SWE-Bench Pro, Scale AI, OpenReview 9R2iUHhVfr]。这意味着：一个真实的 coding agent 任务，要发生几十到几百次 "LLM 推理 + 工具调用" 的往返。

### 要点2：在这个循环里，工具执行（不是 LLM）吃掉了一半以上的时间

在 LLM-工具的串行循环中，工具执行（编译、测试、文件读写、web 抓取）**占总请求时间的 35%-61%**，按领域平均：编码任务 60%、深度研究 50%、科研任务 36% [来源: PASTE: Act While Thinking, arXiv 2603.18897, 2026, §2.2.1，基于 SWE-bench / DeepResearchBench / ScholarQA，Gemini-2.5/GPT-5.2/Qwen 实测]。这意味着：即使在 Agent 里，LLM 推理也不是唯一瓶颈，工具执行这个 "非 LLM" 的部分同样主导着端到端延迟。

### 要点3：通信成本随规模上升——从单 Agent 到多 Agent 的关键转折

单 Agent 的瓶颈在 loop 内部效率（工具调用、本地推理、序列化）。但当多个 Agent 协作时，瓶颈变成 **多个 loop 之间的协调开销**——即"通信"。关键数据：多 Agent 比 Agent ~**2-5×** token/成本（经验法则，⚠️ 数量级估算，受任务复杂度、Agent 数量、交互轮数影响）[来源: 承袭旧详细文档 P2-3，综合 MetaGPT / UIUC 研究 arXiv:2505.18286]；Continuum (arXiv:2511.02230) 测得多 Agent 系统调度气泡可达 **~58%**（⚠️ 指任务间隙，非工具调用时间）。**诊断论断：从单 Agent 到多 Agent，通信占比随规模上升、逐渐压过计算成为主导成本。**

### 要点4：ARM 平台的融合机会——不只是算子加速器

我们的 ARM 平台——**现役 NEON（鲲鹏 920 已有）+ SVE/SME（鲲鹏 930+/950+ 路线图）**——它不只是传统意义上的算子加速器，更是能把"通信的数据搬运/序列化"与"计算的向量化/矩阵化"放进**同一组执行单元**的融合点。**NEON 现在就能做基础融合（向量化序列化/pack/基础 SIMD）**；SVE 的 predicate+gather/scatter 与 SME 的矩阵引擎是"下一站"。**通信计算融合，就是我们在 Agent 时代的主场。**

### 要点5：三部分议程——从 problem setup 到 diagnosis 到 prescription

- **第一部分（基础）**：Agent 执行流程 = 计算↔通信的反复交织（8min）—— 让听众把 agent loop 看成"计算↔通信的反复交织"
- **第二部分（洞察）**：从单 Agent 到多 Agent，通信随规模上升、渐成主导（10min）—— 现状盘点 + 数据论证
- **第三部分（融合）**：ARM 平台（NEON 现役 + SVE/SME 前瞻）的通信计算融合 = 主场机会（9min）—— 武器、论点、算子侧、通信侧、路线图

## 关键数据速览

- SWE-bench Verified 单实例上限 **250 步 / 3 美元** [来源: arXiv 2511.13646] ✅
- 工具执行占 agent 总请求时间 **35%-61%**（编码任务高达 60%）[来源: PASTE, arXiv 2603.18897] ✅
- 多 Agent 比 Agent ~**2-5×** token/成本（⚠️ 数量级估算，受任务/AI数量/轮数影响）[来源: 承袭旧详细文档 P2-3]
- Continuum 调度气泡可达 **~58%**（⚠️ 指任务间隙）[来源: Continuum, arXiv:2511.02230]

## 演讲备注 / 过渡语

> "各位好。在开始之前，先把边界划清楚：今天我们不聊怎么训模型、不聊怎么优化单个 token 的推理。我们聊的是——当 LLM 从一个'被问一次答一次'的东西，变成一个会自己写代码、跑测试、查资料的'循环'之后，整个系统的成本结构发生了什么变化。
> 你可能会惊讶：在一个 coding agent 的请求里，真正花在'跑工具'——编译、测试、读写文件——上的时间，占了 35% 到 61%。在编码任务里，这个比例高达 60%。换句话说，一大半的延迟，根本不在 LLM 身上，而在我们系统人最熟悉的那些地方——进程、IPC、序列化、并发。这就是我们今天的主场。
> 更关键的是：从单 Agent 到多 Agent，通信成本会随规模上升，逐渐压过计算成为主导成本。而在我们的 ARM 平台上——现役 NEON 已经能做基础融合，SVE/SME 是下一站——通信计算融合，就是我们在 Agent 时代的主场。
> 今天分三部分：第一部分看 Agent 执行流程的本质；第二部分看从单到多 Agent 的成本结构怎么变；第三部分看 ARM 平台的融合机会。让我们从第一部分开始。"

========== PAGE 2: P1-1 章节封面 + 划边界 ==========

## 核心观点（一句话）

**Agent 运行时 = loop 里被 LLM 光环掩盖的系统层。** 我们不碰训练/推理内核，聚焦的是：感知→推理→规划→工具调用→记忆更新 这个长循环里的 CPU 调度、内存访问、工具通信、资源瓶颈——这正是我们的主场。

## 详细要点

### 要点1：一句话定义"Agent 运行时"

Agent 运行时，不是 LLM 本身的训练/推理，而是 **当 LLM 被放进一个循环里、反复调用工具、反复进出进程边界时，整个端到端的延迟和成本结构**。我们的主场，是这个循环里那些被 LLM 光环掩盖的系统层：CPU 调度、内存访问、工具通信、资源瓶颈。

### 要点2：边界划定——我们不碰什么

我们不讨论：pre-training / fine-tuning、GPU 利用率优化、FlashAttention / PagedAttention 这种单次推理内核优化（虽然它们很重要：FlashAttention 把 HBM 读写从 O(N²) 降到 O(N)，带来约 **2-4×** 加速 [来源: FlashAttention, NeurIPS 2022, Dao et al., arXiv 2205.14135] ✅；PagedAttention 让 vLLM 吞吐提升 **2-4×** [来源: Efficient Memory Management for LLM Serving with PagedAttention, SOSP 2023, Kwon et al., arXiv 2309.06180] ✅）。我们关心的是：Agent loop 里那些"非 LLM"的部分。

### 要点3：边界划定——我们聚焦什么

Agent = 感知→推理→规划→工具调用→记忆更新的长循环，**不是单次 API 调用**。我们的主场，是这个循环里的：
- **CPU 调度**：进程启动、prefork、资源池化
- **内存访问**：向量检索（gather、距离计算）、小批量计算
- **工具通信**：IPC（HTTP/UDS/共享内存）、序列化（JSON/Protobuf/FlatBuffers）
- **资源瓶颈**：锁竞争、负载不均、fail-slow

### 要点4：本章目标——结束时听众把 agent loop 看成什么

结束时，听众应该把 Agent loop 看成 **"计算↔通信的反复交织"**：
- 每一步是不同计算形态（文本处理、LLM 推理、规划、工具调用、向量检索）
- 本地计算 ↔ 跨边界通信（API 调用/IPC/工具执行）的乒乓
- 混合/突发/长尾/依赖复杂——新 workload 的本质

### 要点5：主场定位——为什么这是系统软件的主场

当循环拉长、工具调用变密集，系统层的东西开始决定胜负：每次启动一个工具子进程的开销、每次跨进程传数据的 IPC 延迟、每次序列化上下文的成本、每次锁竞争和共识延迟。这些恰恰是经典系统工程的领域——**GEMM/Attention + MPI/UCX/SDMA**，我们的能力直接映射。

## 关键数据速览

- FlashAttention **2-4×** 加速 [来源: arXiv 2205.14135] ✅
- PagedAttention 吞吐 **2-4×** 提升 [来源: arXiv 2309.06180] ✅
- 工具执行占 agent 总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅

## 演讲备注 / 过渡语

> "现在我们把边界划清楚：今天我们不聊怎么训模型、不聊怎么优化 GPU 上的单个 token 推理。我们聊的是——当 LLM 从一个'被问一次答一次'的东西，变成一个会自己写代码、跑测试、查资料的'循环'之后，整个系统的成本结构发生了什么变化。
> 这个循环里有五个环节：感知、推理、规划、工具调用、记忆更新。我们的主场，是这个循环里那些被 LLM 光环掩盖的系统层——进程、IPC、序列化、并发。这些恰恰是经典系统工程的领域，也是我们 GEMM/Attention + MPI/UCX/SDMA 能力直接映射的地方。
> 下一页，我们把这个循环拆开看：它到底是什么样的一个 loop？"

========== PAGE 3: P1-2 Agent 执行流程 = 一个循环 ==========

## 核心观点（一句话）

**Agent 不是"一次 API"，而是长循环。** 感知→推理→规划→工具调用→记忆更新，这个 loop 会重复几十到几百次（SWE-bench 上限 250 步/$3），每一步触发不同的计算形态——混合/突发/长尾/依赖复杂，这是新 workload 的本质。

## 详细要点

### 要点1：标准循环图——感知→推理→规划→工具调用→记忆更新

Agent 的标准工作流来自 ReAct 范式：每一轮产生 **一个 Thought（推理）、一个 Action（工具调用）、一个 Observation（工具返回）** [来源: ReAct: Synergizing Reasoning and Acting in LLMs, arXiv 2210.03629]。后续 ASE 2025 的轨迹研究系统统计了多个 SOTA agent 的 TAR（Thought-Action-Result）步数分布 [来源: A Study of Thought-Action-Result Trajectories, ASE 2025]。完整循环：**感知（输入处理）→ 推理（LLM 调用）→ 规划（下一步决策）→ 工具调用（执行）→ 记忆更新（上下文累积）→ 回到感知**。

### 要点2：不是"一次 API"而是长循环——SWE-bench 250 步/$3 上限

传统 LLM 调用是一次 request、一次 response。Agent 是把这个调用放进一个可能长达几十步、上百步的循环里。权威数据：SWE-bench Verified 的标准预算上限是每实例 **最多 250 步、最多 3 美元** [来源: Can Software Engineering Agents Self-Evolve on the Fly? arXiv 2511.13646, 引用 SWE-bench Verified 官方配置] ✅；SWE-bench Pro (Scale AI) 给每个 agent 任务 **最多 50 次工具调用**，作者称该额度 "相当标准" [来源: SWE-Bench Pro, Scale AI, OpenReview 9R2iUHhVfr]。也就是说，一个真实的 coding agent 任务，要发生几十到几百次 "LLM 推理 + 工具调用" 的往返。

### 要点3：每一步触发不同的计算形态——循环的异构性

这个循环不是重复同样的计算，每一步都是不同的形态：
- **感知**：文本处理（tokenization、模板渲染）→ CPU 密集、内存带宽敏感
- **推理**：LLM API 调用（远程、高延迟）→ 本地 CPU 空闲窗口
- **规划**：轻量推理/规则匹配 → 计算轻、逻辑复杂
- **工具调用**：重计算（代码分析、profiler）+ IO（文件读写、网络请求）→ CPU/IO 混合
- **记忆更新**：向量检索（gather、距离计算）+ rerank → 内存随机访问、小批量计算

### 要点4：循环特点——混合/突发/长尾/依赖复杂

这是 Agent 区别于传统推理服务的关键特征：
- **混合**：同一次执行里密集计算（工具）与高延迟通信（API）交替出现
- **突发**：工具调用时间分布极不均匀（成功 12 步/$1.21 vs 全部 21 步/$2.52，SWE-agent 数据）
- **长尾**：长链路任务的尾部少量步骤占用不成比例的资源
- **依赖复杂**：工具输出依赖、状态共享、fail-slow（单步慢拖累全局）

### 要点5：新 workload 的本质——传统调度假设失效

面向单一计算类型的调度器无法处理这种 compute/comm 混合模式。Devin **72% 通过的测试耗时 >10 分钟** [来源: Cognition SWE-bench Technical Report] ⚠️ 数量级估算（vendor self-report，随任务/时间变化）。这证明了：Agent 是一种全新的、混合/突发/长尾的 workload，传统调度假设失效。

## 关键数据速览

- SWE-bench Verified 单实例上限 **250 步 / 3 美元** [来源: arXiv 2511.13646] ✅
- SWE-bench Pro **50 次工具调用**/任务 [来源: Scale AI, OpenReview 9R2iUHhVfr] ✅
- SWE-agent：成功 **12 步/$1.21**，全部（含失败）**21 步/$2.52** [来源: SWE-agent, NeurIPS 2024, arXiv 2405.15793] ✅
- Devin **72%** 的成功任务耗时 **>10 分钟** [来源: Cognition SWE-bench Technical Report] ⚠️ 数量级估算（vendor self-report，随任务/时间变化）

## 演讲备注 / 过渡语

> "现在我们把 Agent 的工作流画出来。看起来很复杂，其实就是五个字一个圈：感知、推理、规划、工具调用、记忆更新。但有两个特征决定了它和普通推理服务完全不一样。
> 第一，它不是一次 API 调用，而是长循环。SWE-bench 的标准预算上限是 250 步、3 美元。一个真实的 coding agent 任务，要发生几十到几百次'LLM 推理 + 工具调用'的往返。
> 第二，每一步都是不同的计算形态——文本处理、LLM 推理、规划、代码分析、向量检索。这导致了混合/突发/长尾/依赖复杂的负载特征。传统面向单一计算类型的调度器，根本处理不了这种 compute/comm 混合模式。
> 下一页，我们把每一步的计算形态拆开看：它们分别是什么计算？分别消耗什么资源？"

========== PAGE 4: P1-3 拆开循环：每步是什么计算形态 ==========

## 核心观点（一句话）

**循环的每一步，是不同的计算形态，对应不同的资源需求。** 感知（CPU 密集）、推理（远程高延迟→本地 CPU 空闲）、规划（计算轻逻辑复杂）、工具调用（CPU/IO 混合）、记忆更新（内存随机访问/小批量计算）——这是异构负载的根源。

## 详细要点

### 要点1：感知——文本处理（tokenization、模板渲染）

**计算形态**：文本处理、tokenization、模板渲染、JSON 序列化/解析
**资源需求**：CPU 密集、内存带宽敏感
**典型延迟**：JSON 解析（simdjson 单核 GB/s 级）[来源: simdjson 官方/GitHub, 2024] ✅；模板渲染（Jinja2 类引擎）通常毫秒级
**优化机会**：SIMD 加速 JSON 解析（simdjson）、模板预编译、零拷贝格式（FlatBuffers）

### 要点2：推理——LLM API 调用（远程、高延迟）

**计算形态**：远程 LLM API 调用（GPT-4o/Claude/DeepSeek 等）
**资源需求**：网络 I/O 密集，本地 CPU 空闲
**典型延迟**：GPT-4o 平均总响应时间约 **570ms**（TTFT≈562ms）[来源: AILatency 持续监控, 2024; Artificial Analysis, 2024] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）；Claude 3.5 Sonnet 约 **671ms**[来源: AILatency; Artificial Analysis, 2024] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）；GPT-4o Mini TTFT 约 **200-400ms**（正常负载）[来源: SitePoint, 2025] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）；DeepSeek 官方 API 平均响应约 **23.64s/请求**（V4-Pro TTFT 最高可达 **128.46s**）[来源: Medium OpenAI vs DeepSeek 对比, 2025; DeepInfra, 2025] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）

### 要点3：规划——轻量推理/规则匹配

**计算形态**：轻量推理、规则匹配、决策树、状态机
**资源需求**：计算轻、逻辑复杂、CPU 缓存敏感
**典型延迟**：通常微秒到毫秒级（取决于规则复杂度）
**优化机会**：规则引擎 JIT 编译、决策缓存、向量化规则匹配

### 要点4：工具调用——重计算（代码分析、profiler）+ IO（文件读写、网络请求）

**计算形态**：重计算（代码分析、profiler、静态分析、Linter）+ IO（文件读写、网络请求、git 操作）
**资源需求**：CPU/IO 混合、内存带宽敏感、可能触发 fork/exec
**典型延迟**：编译/测试（秒级到分钟级）、文件读写（毫秒级）、网络请求（毫秒到秒级）
**关键数据**：工具执行占总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅；fork 1GB 进程平均 **6.5ms**，50GB 进程平均 **253.9ms**[来源: On-demand-fork: A Microsecond Fork, EuroSys 2021, Purdue University] ✅

### 要点5：记忆更新——向量检索（gather、距离计算）+ rerank

**计算形态**：向量检索（embedding、向量数据库 gather、距离计算）、rerank（cross-encoder）
**资源需求**：内存随机访问密集、小批量计算、内存带宽敏感
**典型延迟**：all-MiniLM-L6-v2 约 **4.2ms/句**（~14,200 句/分钟）[来源: AI Tinkerers Lausanne, 2023; SBERT 官方文档, 2024] ⚠️ 数量级估算（environment-dependent）；cross-encoder reranker 单文档 **5-50ms**，100 候选可达 **500ms-5s**[来源: OneUptime Cross-Encoder Reranking, 2026] ⚠️ 数量级估算
**优化机会**：CPU 量化（ONNX INT8 **3.08×** 加速 [来源: SBERT Efficiency 文档, 2025] ⚠️ 数量级估算，environment-dependent）、SIMD 向量化、SVE gather/scatter、SME 矩阵引擎

========== PAGE 5: P1-4 loop = 计算↔通信的反复交织 —— 🎯 主线第一次点题 ==========

## 核心观点（一句话）

**🎯 Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 把"感知→推理→规划→工具调用→记忆"重画成"本地计算 ↔ 跨边界通信"的乒乓：API 等待期间本地 CPU 利用率~0%（串行流水线的灰色空闲）——这是优化的核心靶点。

## 详细要点

### 要点1：重画循环——从五步到乒乓

把 Agent 的标准循环（感知→推理→规划→工具调用→记忆更新）重画成 **"本地计算 ↔ 跨边界通信"** 的乒乓：
- **本地计算**：感知（文本处理）、规划（规则匹配）、记忆更新（向量检索）、工具调用的本地预处理/后处理
- **跨边界通信**：推理（LLM API 调用）、工具调用（远程执行、IPC、文件/网络 IO）

### 要点2：跨边界通信 = API 调用/IPC/工具执行

**跨边界通信**包括：
- **API 调用**：远程 LLM API（GPT-4o/Claude/DeepSeek）
- **IPC**：进程间通信（HTTP localhost、Unix Domain Socket、共享内存）
- **工具执行**：编译、测试、文件读写、网络请求

这些通信的典型延迟：API 调用 **500ms-5s+**（⚠️ 数量级估算，取决于提供商/负载/输出长度）；HTTP localhost **0.1-1ms**（100-1000µs）[来源: Medium — Beyond HTTP: Unix Domain Sockets, 2023] ⚠️ 数量级估算（公开博客实测，随环境变化）；Unix Domain Socket 约 **130µs**，TCP loopback 约 **334µs**[来源: NodeVibe, 2024] ⚠️ 数量级估算（公开博客实测，随环境变化）；共享内存 P99 约 **850ns**[来源: HowTech, 2023] ⚠️ 数量级估算（公开博客实测，随环境变化）。

在一次 LLM API 调用期间（**500ms-5s+**），Agent 主机的 CPU 占用接近 **~0%**（⚠️ 数量级估算，串行阻塞场景，取决于本地是否有其他任务）——这是被同步阻塞白白浪费的算力。主流大模型 API 延迟：GPT-4o 约 **570ms**（TTFT≈562ms）[来源: AILatency 持续监控, 2024] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）；Claude 3.5 Sonnet 约 **671ms**[来源: AILatency; Artificial Analysis, 2024] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）；GPT-4o Mini TTFT 约 **200-400ms**[来源: SitePoint, 2025] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）；DeepSeek 官方 API 平均响应约 **23.64s/请求**[来源: Medium OpenAI vs DeepSeek 对比, 2025] ⚠️ 数量级估算（公开监控数据，随负载/时间变化）。

### 要点4：串行流水线的灰色空闲——优化的核心靶点

当前 Agent 框架普遍采用同步串行循环：等待 API → 取结果 → 做后处理 → 组装下个 Prompt → 等待 API。这种模式下，本地 CPU 在 API 等待期间基本空闲——**灰色空闲区域就是优化的核心靶点**。反向验证：PASTE 通过推测式提前执行工具（在 LLM 思考时并行执行下一个工具），将平均任务完成时间降低 **48.5%**，工具执行吞吐提升 **1.8×**，工具等待时间（stall）减少 **67%**[来源: PASTE, arXiv 2603.18897, §7.2-7.3, 2026] ✅。

### 要点5：🎯 主线第一次点题——Agent loop 本质是 compute↔comm 交织

**Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 从单 Agent 的"工具调用往返"（本地计算 ↔ API 调用），到多 Agent 的"A2A 协作"（多个 loop 之间的消息传递），这个 loop 的本质就是 **compute ↔ comm 的反复交织**。通信占比随规模上升、逐渐压过计算成为主导成本——这是 P2 要论证的核心命题。

========== PAGE 6: P1-5 为什么是"新"workload ==========

## 核心观点（一句话）

**Agent 是混合/突发/长尾/依赖复杂的新 workload。** 传统面向单一计算类型的调度器无法处理 compute/comm 混合模式；SWE-agent 成功 12 步/$1.21 vs 全部 21 步/$2.52 证明突发性——任务时长分布极不均匀，fail-slow 现象显著。

## 详细要点

### 要点1：混合——同一次执行里密集计算与高延迟通信交替出现

同一次 Agent 执行里，密集计算（工具调用：代码分析、profiler、编译、测试）与高延迟通信（API 调用、网络请求、文件 IO）剧烈交替。这是 Agent 区别于传统推理服务的关键：传统推理服务是单一计算形态（矩阵乘），Agent 是 **compute + comm 混合**。工具执行占总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅。

### 要点2：突发——工具调用时间分布极不均匀

**SWE-agent 实测数据**：成功解决的实例中位数 **12 步/$1.21**，全部实例（含失败）均值 **21 步/$2.52**——失败实例消耗的步骤/成本约为成功的近 **2 倍**（fail-slow）[来源: SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering, NeurIPS 2024, arXiv 2405.15793] ✅。这证明：Agent 任务时长分布极不均匀，成功的任务可能 12 步完成，失败的任务可能拖到 21 步甚至更多——**突发性是调度噩梦**。

### 要点3：长尾——长链路任务的尾部少量步骤占用不成比例的资源

长链路 Agent 任务（如复杂 bug 修复、大规模代码重构）的尾部少量步骤占用不成比例的资源。Devin **72%** 通过的测试耗时 **>10 分钟**[来源: Cognition SWE-bench Technical Report] ⚠️ 数量级估算（vendor self-report，随任务/时间变化）。长尾效应导致：少数慢任务拖累整体吞吐，尾部延迟主导用户体验。

### 要点4：依赖复杂——工具输出依赖、状态共享、fail-slow

Agent 任务有复杂的依赖关系：
- **工具输出依赖**：下一步的输入依赖上一步工具的输出（编辑-验证循环：55% 的文件编辑后立即跟随终端调用 [来源: PASTE, arXiv 2603.18897, §2.3.1] ✅）
- **状态共享**：多个 Agent 或工具之间共享上下文、记忆、文件系统
- **fail-slow**：单步慢拖累全局（21 步任务中任意一步超时，整个任务延迟飙升）

这种依赖复杂度导致：**推测执行的覆盖有限**（只能预测下一步，不能跳过依赖）、**负载均衡困难**（无法提前知道任务会跑多久）。

### 要点5：传统调度假设失效——面向单一计算类型的调度器无法处理

传统调度器假设：
- **单一计算类型**：所有任务都是同一种计算（如矩阵乘）
- **可预测时长**：每个任务的执行时间相对稳定
- **独立任务**：任务之间依赖弱，可随意重排序

Agent workload 打破所有这些假设：
- **混合计算类型**：compute + comm 混合，每一步资源需求不同
- **不可预测时长**：12 步 vs 21 步，分布极不均匀（SWE-agent 数据）
- **强依赖**：工具输出依赖、状态共享，fail-slow 显著

**结论**：传统面向单一计算类型的调度器（如 HPC 作业调度、传统推理服务调度）无法处理 Agent 的 compute/comm 混合模式。**这为 P2 的多 Agent 协同洞察铺路**——当多个 Agent 协作时，调度问题会更复杂。

## 关键数据速览

- SWE-agent：成功 **12 步/$1.21**，全部（含失败）**21 步/$2.52** [来源: SWE-agent, NeurIPS 2024, arXiv 2405.15793] ✅
- 工具执行占 agent 总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅
- 55% 的文件编辑后立即跟随终端调用（Edit-Verify 模式）[来源: PASTE, arXiv 2603.18897, §2.3.1] ✅
- Devin **72%** 的成功任务耗时 **>10 分钟**[来源: Cognition SWE-bench Technical Report] ⚠️ 数量级估算（vendor self-report，随任务/时间变化）

========== PAGE 7: P2-1 章节封面 + 诊断问题 ==========

## 核心观点（一句话）

**诊断问题：从 1 个 agent 到 N 个 agent，成本结构怎么变？** 单 Agent 的瓶颈在 loop 内部效率（工具调用、本地推理、序列化）；多 Agent 的瓶颈变成 **多个 loop 之间的协调开销——即"通信"**。本章诊断论断：**通信占比随规模上升、逐渐压过计算成为主导成本。**

## 详细要点

### 要点1：诊断问题——单 Agent → 多 Agent，成本结构怎么变？

P1 我们把单 Agent loop 看成 "计算↔通信的反复交织"，并指出单 Agent 瓶颈在 loop 内部效率（工具执行占 35%-61%、API 等待期间本地 CPU~0%）。现在的问题：当我们从 1 个 agent 扩展到 N 个 agent（A2A 协作），这个成本结构会发生什么变化？哪些成本被放大？哪些原本不存在的成本被引入？

### 要点2：引出新变量——"通信"= 多 loop 之间的协调开销

多 Agent 系统引入了一个单 Agent 不存在的新成本变量：**多个 loop 之间的协调开销——即"通信"**。它包括：Agent 之间的消息传递、状态同步、角色调度、共识达成、负载均衡。单 Agent 的 loop 是 "本地计算 ↔ 跨边界通信" 的乒乓；多 Agent 是 **"多个 loop 各自乒乓 + loop 之间互相通信"**——多了一层 loop 间通信维度。

### 要点3：诊断论断预告——通信占比随规模上升、渐成主导

本章要论证的核心命题：**从单 Agent 到多 Agent，通信占比随规模上升、逐渐压过计算成为主导成本。** 这不是凭直觉的断言，而是有学术基准（UIUC、Continuum）、框架拓扑（AutoGen/MetaGPT/CrewAI/LangGraph）、协议趋势（Google A2A）三类证据支撑的诊断。

### 要点4：本章结构——现状盘点 → 关键数据 → 我们的洞察 → 诊断合成 → 桥到 P3

========== PAGE 8: P2-2 现状盘点：A2A 正在发生什么 ==========

## 核心观点（一句话）

**A2A 生态分三层：框架通信拓扑各异、Google A2A 协议标准化跨厂商互操作、学术基准量化真实成本。** 对我们 CPU 团队的 load-bearing 洞察：**A2A 是纯 Web 传输（HTTP+JSON-RPC+SSE）——CPU 处理 TLS/JSON，无需特殊硬件；真正的性能痛点在服务层 prefill 膨胀和缓存重调度气泡，而非 agent 逻辑本身。**

## 详细要点

### 要点1：主流多 Agent 框架——通信拓扑"各自为政"

四大主流框架的通信机制截然不同，这是 A2A 生态碎片化的根源 [来源: AutoGen/MetaGPT/CrewAI/LangGraph 官方文档] ✅：
- **AutoGen（Microsoft）**：广播/共享 topic 的 **GroupChat 拓扑**——所有 agent 向同一个公共消息线程发布/订阅（pub/sub），由一个 **GroupChatManager** agent 负责选下一发言者（round-robin 或 LLM 选择器）[来源: AutoGen docs — Group Chat design pattern] ✅
- **MetaGPT（ICLR 2024）**：全局共享消息池 + SOPs（标准操作程序）。分配 **5 个顺序角色——Product Manager → Architect → Project Manager → Engineer → QA Engineer**——他们交接结构化输出（PRD、设计文档、接口规格、测试用例）而非自由形式对话；inter-agent 通信流经全局共享消息池（发布-订阅），而非成对聊天链接 [来源: MetaGPT, arXiv:2308.00352, ICLR 2024] ✅
- **CrewAI**：基于角色的**层级任务委托**（hierarchical delegation）拓扑 [来源: CrewAI 官方文档] ✅
- **LangGraph**：把多 agent 工作流建模为**有状态有向图（节点 = agent/工具，边 = 状态转移）**而非自由对话，内置 checkpointer 在每个 super-step 保存状态快照，支持持久化、断点续跑、human-in-the-loop [来源: LangChain 官方 LangGraph 文档] ✅

### 要点2：Google A2A 协议（2025-04-09 宣布，50+ 合作伙伴）——标准化跨厂商互操作

Google 于 **2025 年 4 月 9 日**在 Cloud Next 发布开放协议 **Agent2Agent (A2A)**，由 **50+ 家技术合作伙伴**（Atlassian、Salesforce、SAP、ServiceNow、LangChain 等）及十余家服务商共同参与 [来源: Google Developers Blog — Announcing the A2A Protocol, https://developers.googleblog.com/en/a2a-a-new-era-of-agent-interoperability/] ✅。
- **传输栈**：纯 **HTTP + JSON-RPC 2.0 + SSE（Server-Sent Events）**——应用层协议运行于 HTTPS 之上，不假设 CPU/硬件加速
- **Agent Card**：JSON 能力发现文档，位于 **`/.well-known/agent-card.json`**
- **有状态 Task 对象**：约 **8 个生命周期状态**（submitted / working / input-required / completed，加上 failed / canceled / rejected / auth-required）
- **JSON-RPC 方法（v1.0，PascalCase）**：**SendMessage / SendStreamingMessage / GetTask**（注：旧版 tasks/send 等已重命名，勿引用）[来源: Google Developers Blog — A2A 'How it works'] ✅
- **与 MCP 关系**：A2A **补充（而非替代）** Anthropic 的 MCP（Model Context Protocol）

### 要点3：🎯 CPU 团队的 load-bearing 洞察——A2A 是纯 Web 传输，痛点在服务层而非 agent 逻辑

对我们 ARM/CPU 团队而言，关键洞察是：**A2A 是纯 Web 传输——CPU 处理 TLS/JSON，无需特殊硬件**。真正的性能痛点不在 agent 逻辑本身，而在 **服务层 prefill 膨胀** 和 **缓存重调度气泡** [来源: 综合 UIUC arXiv:2505.18286 + Continuum arXiv:2511.02230，校验文档 P2-2 load-bearing insight] ✅。这意味着：A2A 的"通信"成本，本质是 CPU 上的 JSON 序列化/解析、TLS 握手、HTTP 栈往返——这正是我们 MPI/UCX/SDMA + SIMD 能力的主场。

### 要点4：学术基准开始量化多 Agent 真实成本

- **UIUC 研究（arXiv:2505.18286）**：在 7 个数据集、6 个模型上测得 **多 agent 系统消耗单 agent 等效物约 4×-220× 更多 prefill（输入）token**（⚠️ 数量级估算；220× 极端值完全由最难数据集 AIME 驱动，排除 AIME 后典型 prefill 乘数在 ~5-35× 范围）；decode token 也多 2×-12× [来源: Gao et al., 'Single-agent or Multi-agent Systems? Why Not Both?', arXiv:2505.18286] ✅
- **Continuum（ICLR 2026）**：形式化多轮 LLM agent 中的 **"scheduling bubble"（调度气泡）**——工具调用返回但请求在 GPU 等待队列中等待 KV-cache 重新驻留时的空闲间隙——并引入 KV-cache TTL 机制削减多轮 agent 作业完成时间 [来源: Continuum, arXiv:2511.02230, ICLR 2026] ✅

### 要点5：一张"地图"——建立 A2A 正在发生的共同认知

三层叠加得到一张 A2A 现状地图：**框架生态（拓扑各异）+ 协议趋势（A2A 标准化互操作）+ 学术基准（成本量化）**。共同认知：A2A 正在从"各自为政的框架"走向"标准化协议 + 可量化成本"，而成本的主要来源是 prefill 膨胀和调度气泡——这两者都落在服务层/CPU 侧。

========== PAGE 10: P2-4 我们的洞察①：A2A 数据怎么流 ==========

## 核心观点（一句话）

**A2A 数据流有四种通信模式，序列化是隐藏的税，通信复杂度随 agent 数 O(N²) 爆炸。** Agent 消息的特征是"频繁、小 payload、低延迟敏感"——这对通信机制要求极高，也正是通信成本最容易藏身的地方。主线触点：通信成本在哪。

## 详细要点

### 要点1：四种通信模式——点对点 / 广播 / 层级编排 / 共享黑板

多 Agent 系统的通信可归为四种拓扑模式 [来源: 承袭旧详细文档 P2-4，综合 AutoGen/MetaGPT/CrewAI/LangGraph 官方文档]：
- **点对点（1-1）**：两个 agent 直接消息传递（如 LangGraph 的节点-边直连）
- **广播（1-N）**：一个 agent 向所有其他 agent 发同一消息（如 AutoGen GroupChat 的广播 topic）
- **层级编排（树状）**：supervisor 向多个 worker 委托任务（如 CrewAI 层级委托、LangGraph supervisor 架构）
- **共享黑板（N-1 共享状态）**：多个 agent 通过共享消息池/状态读写（如 MetaGPT 全局消息池）

不同模式对序列化、同步、负载均衡的要求截然不同——这是为什么框架拓扑"各自为政"会带来性能差异。

### 要点2：消息序列化开销——JSON 比 Protobuf 慢约 5-10×（⚠️ 数量级）

Agent 之间和 agent 与工具之间的消息绝大多数是文本/JSON 形状，序列化是隐藏的税 [来源: 承袭旧详细文档 P2-4 + FlatBuffers/Zuplo/simdjson]：
- **JSON 比 Protobuf 慢约 5-10×**（⚠️ 数量级估算，实际倍数高度依赖语言/运行时/数据形态；严谨基准多落在 3-6×，后端编译型语言可达 5-10×）[来源: Zuplo, 2024; inomera proto-json-benchmark, 2023] ⚠️ 数量级估算
- **JSON 体积约为 Protobuf 的 6×**（FlatBuffers 基准：JSON 1475B vs Protobuf 228B vs FlatBuffers 344B）[来源: FlatBuffers 官方 Benchmarks, 2024] ⚠️ 数量级估算
- **FlatBuffers 零拷贝读路径**：decode 耗时为 0（指针解引用），100 万次 Decode+Traverse+Dealloc 仅 0.08s vs Protobuf LITE 302s vs RapidJSON 583s [来源: FlatBuffers 官方 Benchmarks, 2024] ✅
- **simdjson**：单核 JSON 解析达 GB/s 级（Minify 6 GB/s、UTF-8 校验 13 GB/s），比 RapidJSON 快 4× 以上 [来源: simdjson VLDB Journal 28(6) 2019, arXiv:1902.08318] ✅

### 要点3：通信复杂度 O(N²) 爆炸——全连接拓扑

如果多 Agent 做成全连接 mesh，通信通道数是 **O(N²)**——agent 数翻倍，通信边数变 4 倍 [来源: 承袭旧详细文档 P2-4]：
- **10 个 agent = 45 条通道**
- **100 个 agent = 4950 条通道**
对比之下，集中式协调（supervisor/hub-and-spoke）是 O(N)，扩展性好得多——这正是 LangGraph supervisor 架构效率提升的原因（P2-2 提及）。

### 要点4：Agent 消息特征——频繁、小 payload、低延迟敏感

Agent 之间的消息与训练/推理的大 batch 矩阵截然不同 [来源: 承袭旧详细文档 P2-4]：
- **频繁**：每一步 Thought-Action-Observation 都可能触发跨 agent 消息（如 MetaGPT 5 角色顺序交接、AutoGen 广播）
- **小 payload**：单条消息通常是结构化文本/JSON（PRD、工具返回、状态更新），不是大张量
- **低延迟敏感**：消息往返延迟直接叠加到端到端 agent 延迟（木桶效应放大）

这意味着：通信机制的 per-message 开销（序列化、IPC 往返、锁）会被高频放大——优化 per-message 延迟比优化 throughput 更关键。

### 要点5：主线触点——通信成本藏在这三处

这一页定位"通信成本在哪"：**(1) 序列化税**（JSON 5-10× 慢、6× 体积）、**(2) O(N²) 通道爆炸**、**(3) per-message 高频放大**。这三处正是 P3 通信侧（SIMD 加速 comm 数据通路、零拷贝 IPC、向量化序列化）要打的目标。

## 关键数据速览

- JSON 比 Protobuf 慢约 **5-10×**（⚠️ 数量级估算，依赖语言/运行时/数据形态，严谨基准多 3-6×）[来源: Zuplo/inomera, 2023-2024] ⚠️ 数量级估算

- 全连接通信复杂度 O(N²)：**10 agents=45** 条通道、**100 agents=4950** 条通道 [来源: 承袭旧详细文档 P2-4] ✅（数量级/拓扑推算）

========== PAGE 11: P2-5 我们的洞察②：被锁与不均吃掉的延迟 ==========

## 核心观点（一句话）

**当多 Agent 共享状态时，锁竞争带来 10-100× 退化（✅ 已确认），分布式共识 Raft/Paxos 引入 ms 级延迟墙，负载不均让最慢 agent 拖垮整体。** 推荐方向：Agent 系统通常可容忍最终一致性，无状态+消息传递模式延迟仅 1-5μs。主线触点：通信成本在哪。

## 详细要点

### 要点1：锁竞争 10-100× 退化（✅ 已确认）

当多 Agent 共享状态依赖传统锁时，锁保护的临界区构成 Amdahl 定律的"串行部分"，高竞争下可造成 **10-100×** 吞吐退化（✅ 已确认，承袭旧详细文档 P2-5；实际倍数取决于临界区长度、线程/核数比、锁类型与工作负载）[来源: ACCU Overload — Refocusing Amdahl's Law, Teodorescu, 2020; EMSE 2017; 承袭旧详细文档 P2-5] ✅：
- **5-10 个 agent 频繁写**：约 **10-30%** 时间花在等锁 [来源: 承袭旧详细文档 P2-5] ⚠️ 数量级估算（随工作负载）
- **10+ 个 agent 高并发写**：**50%+** 时间等锁，吞吐量骤降 [来源: 承袭旧详细文档 P2-5] ⚠️ 数量级估算（随工作负载）

### 要点2：分布式共识延迟墙——Raft/Paxos ms 级（10-50ms，10 节点）

如果多 Agent 需要强一致共识（如共享可写状态），传统共识算法引入 ms 级延迟墙 [来源: 承袭旧详细文档 P2-5 + etcd 官方性能文档] ⚠️ 数量级估算（随部署拓扑/存储介质）：
- **etcd (Raft) LAN 环境**：单次提交延迟约 **1-10ms**，瓶颈是 fdatasync（HDD 约 10ms、SSD 通常 <1ms）[来源: etcd 官方性能文档, 2017] ✅
- **跨可用区部署**：PUT 延迟单区约 69.8ms vs 多区约 72.6ms，地理分布放大 RTT [来源: etcd 官方性能文档; Gardener etcd Network Latency Benchmark, 2023] ⚠️ 数量级估算
- **PBFT**：约数百 ms [来源: 承袭旧详细文档 P2-5] ⚠️ 数量级估算
- 论文按 10 节点规模估算，Raft/Paxos 典型延迟在 **10-50ms** 量级 [来源: 承袭旧详细文档 P2-5] ⚠️ 数量级估算

### 要点3：负载不均衡——最慢 agent 拖累整体（木桶效应）

多 Agent 的负载天然不均 [来源: 承袭旧详细文档 P2-5]：
- **资源画像相反**：编译 agent（CPU 100%）vs 产品 agent（CPU 10%）→ 端到端被编译 agent 决定
- **MetaGPT 角色消融**：协调开销随角色数显著变化（1 agent $0.915 → 4 agents $1.385），证明不同角色负载不均 [来源: MetaGPT (ICLR 2024) Table 3] ✅
- **Google DeepMind**：集中式协调在可并行任务上性能提升约 81%，但前提是负载被合理分派 [来源: Google DeepMind — Towards a Science of Scaling Agent Systems, arXiv:2512.08296, 2025] ⚠️ 数量级估算
- 与 P2-3 木桶效应叠加：最慢 agent 决定端到端，负载不均让"最慢"更频繁出现

### 要点4：推荐方向——最终一致性 + 无状态消息传递

判断：**Agent 系统通常可容忍最终一致性**，不需要银行级强一致 [来源: 承袭旧详细文档 P2-5]：
- **无状态 + 消息传递模式**（Actor / 事件驱动）延迟仅 **1-5μs**（⚠️ 数量级估算，依赖 IPC 机制/拓扑）[来源: 承袭旧详细文档 P2-5] ⚠️ 数量级估算
- **无锁队列**：低线程数（1-8）下 `boost::lockfree::queue` 比 std::mutex 队列吞吐高 **75%-150%** [来源: Qihoo360/evpp benchmark, 2017] ⚠️ 数量级估算
- **MPI 的集合通信**（Allreduce/Broadcast）正是无锁、去中心化 ring/tree 算法的工业级实现——与 Actor 模型同源思想

### 要点5：主线触点——被锁与不均吃掉的延迟

这一页定位"通信成本"的第二处：**锁竞争 10-100× + 共识 ms 级延迟墙 + 负载不均**。三者的共同解药是"无状态 + 消息传递 + 无锁原语"——这正是 MPI/UCX/SDMA 的能力映射，也是 P3 通信侧的优化靶点。

- **融合架构**：在 ARM NEON/SVE/SME 上，序列化/pack/gather/转置本身就是 SIMD/矩阵操作——**同一组单元、同一份布局、无边界拷贝**。

========== PAGE 15: P3-2 NEON（现役）/ SVE（前瞻）/ SME（前瞻）—— ARM 的三类向量/矩阵武器，为什么对 Agent 特别 ==========

## 核心观点（一句话）

**Agent 负载是"变长/不规则/小批量"的**——NEON（现役）已能做基础融合；SVE 的 **predicate + gather/scatter** 与 SME 的 **ZA tile + outer-product 矩阵引擎** 正是为这种不规则负载设计，它们随鲲鹏 930+/950+ 解锁，是融合的"下一站"。

## 详细要点

### 要点1：NEON —— 现役（鲲鹏 920 现在就有）

- **定义**：ARM 的 **Advanced SIMD**，**固定 128-bit 向量宽度**（16 × 8-bit / 8 × 16-bit / 4 × 32-bit / 2 × 64-bit lane）[来源: ARM ARM, Advanced SIMD/NEON；IEEE Micro 2021 "TaiShan V110's SIMD is compatible with NEON-basic instructions"] ✅
- **特性**：**固定宽度**、**整向量操作**、**无 per-lane 谓词**、**无原生 gather/scatter**（仅 LD1/LD2/LD3/LD4 连续/交错结构加载；非连续访问需用 LD1/TBL 等组合，或 vzip/vuzp/vtrn 寄存器级转置）[来源: ARM ARM] ✅
- **位置**：**鲲鹏 920 现役可立即用** [来源: IEEE Micro 2021 + VLDB Journal 2022，见平台文档 §1 表] ✅

### 要点2：SVE / SVE2 —— 前瞻（鲲鹏 930+/950+ 解锁；属 ARMv9-A）

SVE（Scalable Vector Extension）三件套特性 [来源: Stephens et al., "The ARM Scalable Vector Extension (SVE)", IEEE Micro 37(2), 2017, arXiv:1803.06185] ✅：
1. **向量长度可扩展（vector-length agnostic, VLA）**：实现可选 **128–2048 bit，128-bit 步进**，软件与具体宽度解耦——同一二进制跨宽度运行。
2. **Predicate 寄存器**（P0–P15）：**逐 lane 谓词**，对每个元素独立启用/禁用——变长负载天然适配。
3. **原生 gather-load / scatter-store**：向量化**非连续**内存访问——不规则消息字段天然适配。

SVE2（FEAT_SVE2）：SVE 的指令集扩展版，**引入于 ARMv9-A** [来源: ARM Developer "Introducing SVE2", https://developer.arm.com/documentation/102340/0100/Introducing-SVE2] ✅。⚠️ 注意：鲲鹏用华为自研 TaiShan 核（非 Arm Neoverse），930 是否落地 SVE2 **公开未确认**，应作前瞻处理。

### 要点3：SME / SME2 —— 前瞻（鲲鹏 930+/950+ 解锁；属 ARMv9-A 可选扩展）

FEAT_SME（Scalable Matrix Extension），在 SVE 之上的**矩阵扩展**，三要素 [来源: ARM Developer "SME Introduction" / "SME Instructions Part 2"] ✅：
1. **ZA tile 寄存器**（`VL × VL` 的 **2D 数组**，可切分为命名 tile）。
2. **Outer-product 引擎**：对两个 SVE 向量做外积 `C += A ⊗ B`，累加进 ZA tile；**外积之和 = 矩阵乘**。
3. **Streaming-mode SVE**：进入 SME 时切换到 streaming SVE 模式（可有不同向量长度）。
- 核心指令：**MOPA（Matrix Outer Product Add）** [来源: DATE 2026 "KirbyMM", https://date26.date-conference.com/proceedings-archive/2026/DATA/316.pdf] ✅

⚠️ SME 为 ARMv9-A **可选**扩展；鲲鹏 930+ 是否搭载 **公开未确认**，应作前瞻处理 [来源: docs/superpowers/research/2026-06-09-platform-sme-sve.md §1] ✅。

### 要点4：为什么这些特性对 Agent 负载特别（口径落地）

Agent runtime 的负载形态 [来源: 承袭 P1-3/P1-5 + P2-4 + P3-4 论点]：
- **变长**：token/消息/工具输出长度不规则 → **predicate**（SVE）原生逐 lane 谓词；NEON 需软件处理边界、mask 模拟。
- **不规则内存访问**：消息体（JSON/Protobuf/嵌套结构）字段非连续；向量检索的 gather 式访问 → **原生 gather/scatter**（SVE）；NEON 无此指令，需 LD1/TBL 组合拼凑。
- **小批量**：单 agent/小 swarm 的 embedding/rerank/attention 是"瘦高"矩阵（batch 1-8 vs 训练 32-1024）→ **outer-product + ZA tile**（SME）天然适配；NEON 整向量 GEMM 在小矩阵上占满度低。

### 要点5：一句话收束——同一融合主线的现在与下一站

NEON 让**今天**就能在 SIMD 上做融合（向量化序列化/pack/基础转置）；SVE 的 predicate+gather/scatter 与 SME 的矩阵引擎，是"不规则、变长、小批量"agent 负载的**结构性匹配**——它们随鲲鹏 930+/950+ 解锁时，融合会从"基础 SIMD 加速"升级到"predicate/gather-scatter/矩阵引擎级融合"。**前后不是两个方向，而是同一融合主线的现在与下一站。**

## 关键数据速览

- NEON：固定 **128-bit** SIMD（16×8/8×16/4×32/2×64 lane），无 per-lane 谓词、无原生 gather/scatter [来源: ARM ARM；IEEE Micro 2021] ✅（ISA 事实）
- NEON 位置：鲲鹏 920（TaiShan v110, ARMv8.2-A）现役可立即用 [来源: IEEE Micro 2021 + VLDB Journal 2022] ✅（多源交叉确认）
- SVE 向量长度：**128-2048 bit，128-bit 步进**，VLA（一次编写跨宽度运行）[来源: Stephens et al., IEEE Micro 37(2) 2017, arXiv:1803.06185] ✅（架构规范，无需"约"）
- SME 三要素：ZA tile（VL×VL 2D 数组）+ Outer-product（C += A⊗B，外积之和 = 矩阵乘）+ MOPA 指令 [来源: ARM Developer SME blog + DATE 2026 KirbyMM] ✅（ISA 事实）

========== PAGE 16: P3-3 通信计算融合 = 三件事 — 🎯 thesis 展开 ==========

## 核心观点（一句话）

**🎯 thesis 展开——通信计算融合的三件事：①重叠 ②数据搬运即计算 ③kernel+原语协同。** 在现役 NEON 上**今天就能做融合**（向量化序列化/pack/基础 SIMD）；SVE 的 predicate+gather/scatter 与 SME 的矩阵引擎是"下一站"——前后是同一融合主线的现在与未来。

## 详细要点

### 要点1：① 重叠——通信期间做计算（延迟隐藏）

通信期间做计算，是经典的 HPC 延迟隐藏技术，经典实现是 MPI 非阻塞通信（Isend/Irecv），让有用计算与消息传输并发进行，可带来显著的性能提升 [来源: Bernholdt, Nieplocha, Sadayappan, Shet, Tipparaju, "Characterizing Computation-Communication Overlap in Message-Passing Systems", Ohio State/PNNL, OSTI 944757, 2008] ✅。

**对 Agent 的映射**：API 等待 500ms-5s 期间，本地 CPU 不是必须空闲——可用 SIMD 做数据预处理/后处理/向量检索（embeddings、rerank、prompt 模板渲染），把"通信等待"窗口转化为"计算"窗口。

**重叠非免费的告诫**：Georgia Tech 表征（A100/H100/MI250/MI210，GPT-3/LLaMA 训练）实测，重叠计算和通信导致相对孤立计算的**平均 ~18.9% 计算减速（最高 40%）**；反之，完全顺序执行平均比重叠执行**慢 10.2%（最高 26.6%）**——即重叠净为正但不免费 [来源: Lee, Oh, Kim et al., Georgia Tech/KAIST, ISPASS 2025, arXiv:2507.03114] ✅。⚠️ **这是 GPU 特定数据，对 agent-runtime 只能作定性类比**：协同调度消息与工作，以 kernel 级争用换端到端增益——保持定性，不要呈现为 CPU/通用端到端减速乘数。

### 要点2：② 数据搬运即计算——序列化/pack/转置/gather 本身就是向量化/矩阵化工作

"数据搬运即计算"的核心论点：序列化、pack/unpack、memcpy、AoS↔SoA 转置本身**就是可向量化的计算 kernel**，不是无谓的搬运 [来源: 新写，承接 P3-5 校验] ✅。

- **NEON 现役**：simdjson 式 JSON/序列化解析（NEON 后端，GB/s 级）、基础 pack/转置——**今天就能做** [来源: Langdale & Lemire, VLDB Journal 28(6) 2019, arXiv:1902.08318] ✅。
- **SVE/SME 增量**：Open MPI 论文在 Fujitsu A64FX（512-bit SVE）实测，基于 SVE gather/scatter 的 datatype pack/unpack 比非-gather 基线快 **2×**；SVE 向量化 MPI local reduce 比逐元素快 **4×**（VL=512）[来源: Zhong/Shamis(Arm)/Cao/Bosilca/Sumimoto(Fujitsu)/Miura/Dongarra, IEEE CCGrid 2020, https://www.netlib.org/utk/people/JackDongarra/PAPERS/using-arm.pdf] ✅。**限定**：2× 针对 blocklen=1, gap=1 的非连续向量类型；4× 是 vs 未自动向量化的逐元素；**A64FX 特定，非通用/非鲲鹏**。
- **gather/scatter 警告**：ARM SVE gather-load/scatter-store 是规则化非连续访存的指令级原语，但**高度分散模式"相当慢"**（lattice QCD Wilson 核在 A64FX 上的实测结论）[来源: ACM, "Using SVE to Accelerate Lattice QCD", https://dl.acm.org/doi/fullHtml/10.1145/3581576.3581610] ✅。⚠️ 定性——**访存连续性才是决定性因素**，不要默认 gather/scatter 一定赢。

### 要点3：③ kernel+原语协同——comm 原语与算子共享 SIMD 单元+内存布局

comm 原语（序列化、pack、搬运）与算子（向量化 GEMM、attention）**共享同一组 NEON/SVE/SME 执行单元 + 同一份内存布局**，消除两者之间的边界拷贝。这是融合的物理基础：不再是"通信侧一套机制、计算侧另一套机制、边界一次拷贝"，而是"同一组单元、同一份布局、无边界拷贝"。这一论点**在现役 NEON 上即可立论**（序列化/pack 本身就是 128-bit SIMD 操作），SVE/SME 是增量增强（更宽向量、predicate、矩阵引擎）[来源: 新写，承接 P3 spec §5 + 平台文档 §4] ✅。

### 要点4：论点收束——今天就能做 vs 下一站

三件事（重叠 / 数据搬运即计算 / kernel+原语协同）的论点：**在现役 NEON 上今天就能做融合**——向量化序列化/pack/基础 SIMD 是现役鲲鹏 920 立即可做的；SVE 的 predicate+gather/scatter 与 SME 的矩阵引擎是"下一站"——前后是同一融合主线的现在与未来。**零拷贝 IPC**（iceoryx2，常数级 sub-µs、与 payload 无关）是"不搬运载荷"的极端端点（⚠️ 数量级，定性）[来源: iceoryx2 Discussion #435] ⚠️。

## 关键数据速览

- 重叠是经典 HPC 延迟隐藏（MPI Isend/Irecv），带来显著/数量级收益（无固定乘数）[来源: Bernholdt/Shet 等, OSTI 944757, 2008] ✅（定性，无乘数）
- Georgia Tech 重叠代价：重叠导致计算减速**平均 ~18.9%（最高 40%）**；纯顺序比重叠慢**平均 10.2%（最高 26.6%）**（⚠️ GPU 特定 A100/H100/MI250/MI210，CPU 类比定性）[来源: arXiv:2507.03114, ISPASS 2025] ✅
- Open MPI SVE（A64FX, 512-bit）：gather/scatter pack/unpack 比非-gather 基线快 **2×**；SVE local reduce 比逐元素快 **4×**（✅ 测量值；A64FX 特定，非鲲鹏）[来源: Zhong/Shamis/Cao/Bosilca/Sumimoto/Miura/Dongarra, IEEE CCGrid 2020] ✅
- gather/scatter 高度分散模式"相当慢"（⚠️ 定性，访存连续性决定）[来源: ACM Lattice QCD on A64FX] ✅（定性警告）
- iceoryx2 零拷贝 IPC：常数级 sub-µs、与 payload 无关（⚠️ 数量级；勿暗示快于 raw shmem）[来源: iceoryx2 Discussion #435] ⚠️

> "🎯 这一页展开 thesis——通信计算融合，就是三件事。
> 第一，重叠：通信期间做计算。API 等待 500 毫秒到 5 秒，这段时间本地 CPU 不必空闲，可以 SIMD 做向量检索、rerank、模板渲染。这是经典 HPC 思想，MPI 的非阻塞通信几十年前就在做。但要记住重叠不免费——GPU 训练上实测，重叠会让计算 kernel 减速约 19%，但纯顺序更慢、比重叠还慢约 10%，所以净为正。这只是 GPU 数据，对 CPU agent 是定性类比。

========== PAGE 17: P3-4 算子侧：小批量 Agent kernel on ARM SIMD（NEON-now + SME-roadmap） ==========

## 核心观点（一句话）

**Agent 算子（embedding/rerank/小批量 GEMM/attention）是"瘦高"矩阵**，结构性不同于训练大 batch 方阵 GEMM。**NEON 现役**做基础向量化（LibXSMM/LibShaLom 风格小-GEMM kernel）；**SME 前瞻**用 outer-product + ZA tile 适配"瘦高"小批量矩阵——但 SME 不是万能药，瘦/边缘形状需 SME+SVE 混合策略。

## 详细要点

### 要点1：Agent 算子特征——"瘦高"矩阵

Agent 运行时产生小/瘦矩阵工作负载——单 agent 或小群体的 embedding 查找、rerank 评分、每步 attention——batch 通常 **1-8**，远低于训练的 **32-1024**。这使 GEMM 矩阵呈"扁平/瘦"形状，算术强度低、向量通道利用率不足，传统 BLAS（为大方阵批量 GEMM 调优）在这些形状上效率低下 [来源: 新写，承接 P3-4 校验 + FlashDecoding++ 论点] ⚠️。

### 要点2：NEON（现役）机会——基础向量化 + 小-GEMM kernel

128-bit 固定宽度 SIMD 做基础向量化，机会在**专用小-GEMM kernel**（区别于通用 BLAS）：
- **LibXSMM**（SC15, Intel）：通过针对主机 CPUID 的 JIT 代码生成，小 GEMM 相对通用 BLAS/MKL 加速 **>10× 是可达值（上限）**（"depending on the CPU and application"），典型 MKL 对比多为 **2-4×** [来源: LIBXSMM, SC15, Heinecke/Henry/Hutchinson/Pabst, https://dl.acm.org/doi/10.5555/3014904.3015017] ✅。⚠️ 必须带"可达/最多"+"取决于 CPU 与应用"；原论文场景是科学计算（CP2K/SeisSol/NEK5000），类比为 agent embedding/rerank many-small-GEMM 是**合理延伸推断而非论文直接结论**；无公开鲲鹏 920 专用数字。
- **LibShaLom**（SC'21, Yang/Fang/Dong/Su/Wang）：target **ARMv8 多核（含鲲鹏 920）**，核心创新是**降低 packing 开销（小 M 时约占运行时 ~50%）**——通过 (1) 运行时选择性 packing + (2) 微内核内 packing 与 FMA 计算重叠；额外用解析模型（拉格朗日乘子）导出微内核 tile 形状（mr=7, nr=12）[来源: LibShaLom, SC'21, https://jianbinfang.github.io/files/2021-06-22-sc.pdf] ✅。⚠️ 论文无 agent/LLM 内容，agent 类比标⚠️/方向性投影。

### 要点3：SME（前瞻）增量——outer-product + ZA tile 适配"瘦高"小批量

SME 的 outer-product + ZA tile 是为"瘦高"小批量矩阵设计，但**不是万能药**：
- **KirbyMM（DATE 2026）**：SME MOPA 外积提供**峰值 0.512 TFLOPS FP32/核**（不是实测稠密 GEMM 吞吐——实现的稠密 GEMM 仅约 0.33 TFLOPS/核），vs SVE MLA 的 0.128 TFLOPS = **4×** [来源: KirbyMM, DATE 2026, https://date26.date-conference.com/proceedings-archive/2026/DATA/316.pdf] ✅。**但** SME 的高计算把稠密 GEMM 推入**内存带宽受限（roofline）**，所以 SME 对瘦/边缘矩阵**不是万能药**。0.512 必须标"峰值/peak"。
- **KirbyMM 边缘 case**：专用 SME+SVE 混合例程在边缘形状（**17×16×16、19×32×32、35×32×32、19×48×32、18×64×32，FP32，5 个瘦 case**）上**平均比 KBLAS 快 3.59×**、比纯 SME 主例程快 1.36× [来源: KirbyMM, DATE 2026, Fig. 12] ✅ 测量值。**限定**：平台是 **LX2（ARMv9/SME，鲲鹏 920 的继任者核，不是当前 920）**；FP32；五个特定瘦形状；是平均值。
- **MpGEMM（arXiv:2512.21473）**：SME INT8 外积指令是 **SMOPA**（整数外积指令，勿与浮点助记符混淆）——INT8→INT32 整数外积。在 Apple M4 Pro 单线程、四片 ZA tile 全激活时峰值约 **4010 GOPS INT8** vs 其他精度约 2006 GOPS（约 **2×**）[来源: MpGEMM, arXiv:2512.21473, https://arxiv.org/abs/2512.21473] ✅。⚠️ **标"SME 家族行为，需在目标硅上实测验证"**（M4 Pro 数据，非鲲鹏）。

### 要点4：为什么瘦形状低效——FlashDecoding++ 类比（⚠️ GPU 特定，CPU 类比）

FlashDecoding++（MLSys 2024, 最佳论文）实测：LLM 推理中 GEMM 矩阵呈扁平/瘦形状，既往 GPU 库（cuBLAS/CUTLASS）补零后造成**约 50%（>50%）的性能损失** [来源: FlashDecoding++, MLSys 2024, https://mlsys.org/virtual/2024/session/2782] ✅。⚠️ **这是 GPU 特定数据**。对 CPU 只能作**定性/数量级类比**："类似地，CPU 上单 agent 解码的极小 M/N 会使 NEON/SME 向量通道大量闲置（数量级估算，具体损耗取决于向量宽度与算子形状）"——**不要把 50% 当 CPU/NEON-SME 实测数字**。

### 要点5：映射团队能力 + 口径收束

**能力映射**：GEMM/Attention 优化经验 → Agent-native 小批量 kernel 库（NEON 现役小-GEMM kernel + SME 前瞻矩阵引擎混合策略）。**口径**：NEON-now（基础向量化 + LibXSMM/LibShaLom 风格小-GEMM）+ SME-roadmap（outer-product + ZA tile 增量）。⚠️ 无 SME 实测数据标"数量级 + 可自测 micro-benchmark"；SME 时代的胜利需要**混合策略**——对计算受限块用矩阵单元，对瘦/边缘情况回退到 SVE/NEON。

## 关键数据速览

- Agent 算子 batch **1-8**（vs 训练 32-1024）→ "瘦高"矩阵 [来源: 承袭 P3-4 校验] ⚠️
- LibXSMM：小 GEMM 加速 **>10×（可达/上限）**，典型 MKL 对比 **2-4×**（"取决于 CPU 与应用"）[来源: LIBXSMM, SC15] ✅（无鲲鹏 920 专用数字）
- LibShaLom：降低 packing 开销（小 M 约占运行时 ~50%），ARMv8 多核含鲲鹏 920；mr=7/nr=12 解析 tile [来源: LibShaLom, SC'21] ✅（agent 类比⚠️/方向性投影）
- KirbyMM：SME MOPA **峰值 0.512 TFLOPS FP32/核**（稠密 GEMM 实测仅约 0.33）vs SVE MLA 0.128 = **4×**；稠密 GEMM 内存受限 [来源: KirbyMM, DATE 2026] ✅
- KirbyMM 边缘 case：专用 SME+SVE 混合平均比 KBLAS 快 **3.59×**（LX2 ARMv9/SME，5 个瘦 case FP32，非当前 920）[来源: KirbyMM, DATE 2026] ✅
- MpGEMM：SME INT8 外积是 **SMOPA**（整数外积，非浮点助记符），M4 Pro 峰值约 **4010 GOPS** vs 其他精度约 2006（约 2×）[来源: arXiv:2512.21473] ✅（"需在目标硅实测验证"）
- FlashDecoding++：GPU 瘦 GEMM 补零约 **50%** 性能损失（⚠️ GPU 特定，CPU 类比定性）[来源: MLSys 2024] ✅

## 演讲备注 / 过渡语

> "算子侧。Agent 的 embedding、rerank、小批量 attention 是'瘦高'矩阵，batch 通常 1 到 8，和训练的几十上百完全不同。

========== PAGE 18: P3-5 通信侧：SIMD 加速 Comm 数据通路（NEON-now + SVE/SME-roadmap） ==========

## 核心观点（一句话）

**A2A/工具调用流量绝大多数是 JSON 形状**，序列化/验证数据通路是现实瓶颈。**NEON 现役**做 simdjson 式 GB/s JSON 解析（现役鲲鹏 920 立即可做）；**SVE/SME 前瞻**用 predicate/gather-scatter/tile 做增量增强。NEON 无 gather/scatter 是 ISA 事实，非连续访存需寄存器级转置（vzip/vuzp/vtrn）。

## 详细要点

### 要点1：序列化加速——NEON 现役就能做 simdjson 式 GB/s 解析

**simdjson**（Langdale & Lemire, VLDB Journal 28(6) 2019, arXiv:1902.08318）是首个在单商用核上达到 **GB/s 级**完全校验 JSON 解析器；指令数仅为 RapidJSON 的 **1/4 或更少**；约 **4×** 快于 RapidJSON、约 **25×** 快于 nlohmann/json [来源: Langdale & Lemire, VLDB Journal 28(6) 2019, arXiv:1902.08318] ✅。

- **运行时动态派发后端**：ARM 上为 **NEON**（Apple/A 系、M 系）或 **SVE**（Graviton/Ampere/Neoverse 等服务器核）[来源: simdjson README] ✅。
- **ARM 服务器核实测**（find_tweet，单核）：**AWS Graviton 3（Neoverse V1）约 3.6 GB/s、Graviton 4（Neoverse V2）约 4.6 GB/s**，仍落后 Apple M2（约 6.4 GB/s）[来源: Daniel Lemire 博客, https://lemire.me/blog/2024/07/10/benchmarking-arm-processors-graviton-4-graviton-3-and-apple-m2/] ✅。⚠️ **用"Arm V 系列服务器核（Neoverse V1/V2）"，不要暗示鲲鹏 = V2**（鲲鹏用华为自研 TaiShan 核，非 Arm Neoverse）。
- **SVE2 前瞻增量（SVMATCH）**：Neoverse-N2 上 2 cycle/指令，单条指令在 16 字节向量中匹配至多 16 个 token；应用于 Bytedance Sonic JSON 解码器，吞吐提升**区间 -0.3%~24.3%**（citm_catalog +24.3%、twitter +18.8%、github_events +19.2%；otfcc 仅 +3.4%、canada -0.3%）[来源: Arm Architectures & Processors blog, Yibo Cai, 2024-09] ✅。⚠️ **标"最高约 24%"+ 范围**——提升高度依赖语料库，**不要暗示 24% 均匀**；**微基准（Sonic decoder on N2），不是端到端应用数字**。
- ⚠️ **不存在 SVE 版 JSON 解析器的公开 GB/s 数字**（simdjson issue #2683 仍开放）——**不为 SVE JSON 附任何吞吐数字** [来源: 新写，承接 P3-5 校验] ✅。

### 要点2：共享内存零拷贝——iceoryx2（⚠️ 数量级，定性）

对于节点本地 agent IPC，真正的零拷贝共享内存中间件（iceoryx2）通过完全消除序列化达到**常数级 sub-µs 延迟，且与 payload 大小无关**（无序列化、无锁、基于偏移的真零拷贝）[来源: iceoryx2 Discussion #435, github.com/eclipse-iceoryx/iceoryx2/discussions/435] ⚠️。⚠️ **不要暗示 iceoryx2 绝对延迟快于原始 shmem**（同一来源显示它单字节反而慢约 600ns）；"约 100ns 改进"是 benchmark 方法论脚注（warmup 降频），不是产品加速。可防御的标题是定性/数量级："sub-µs、与载荷无关"。

### 要点3：消息搬运加速——SVE 原生 gather/scatter（前瞻）；NEON AoS→SoA 转置（现役）

- **NEON 没有 gather/scatter 指令**（仅 LD1/LD2/LD3/LD4 连续/交错结构加载）——非连续 AoS 字段需用 **vzip/vuzp/vtrn** 做寄存器级转置 [来源: ARM ARM；Stack Overflow "Is there an ARM Neon Gather Instruction?"] ✅ ISA 事实。
- **AoS→SoA 转置**是 canonical SIMD 数据重排手法（ryg blog, SIMD Transposes）；NEON 上用 vzip/vuzp/vtrn intrinsics；SoA + 连续 ld1/st1 是首选 [来源: ryg blog SIMD Transposes 1; Arm doc 102107a] ✅。
- **gather/scatter 低效性定性**：依赖 gather/scatter 的向量化性能通常**远劣于 unit-stride** 加载/存储——保持定性（"远劣于"），不发明精确减速乘数 [来源: Intel Explicit Vector Programming BKM] ⚠️。
- **SVE 原生 gather/scatter** 做非连续消息搬运（前瞻，鲲鹏 930+ 解锁）——是 SVE/SVE2 相对 NEON 的明确架构改进。

### 要点4：映射团队能力 + 口径收束

**能力映射**：MPI/UCX/SDMA → Agent-native 通信原语（零拷贝 IPC、向量化序列化、SIMD AoS↔SoA repack）。**口径**：NEON-now（simdjson 式 GB/s 解析 + vzip/vuzp/vtrn 转置，现役鲲鹏 920 立即可做）+ SVE/SME-roadmap（gather/scatter/predicate/SVMATCH/tile 增量，930+/950+ 解锁）。

## 关键数据速览

- simdjson：首个单商用核 **GB/s 级**完全校验 JSON 解析器；指令数仅 RapidJSON 的 **1/4 或更少**；约 **4×** 快于 RapidJSON、约 **25×** 快于 nlohmann/json [来源: Langdale & Lemire, VLDB Journal 28(6) 2019, arXiv:1902.08318] ✅
- simdjson ARM 服务器核：Graviton 3（Neoverse V1）约 **3.6 GB/s**、Graviton 4（Neoverse V2）约 **4.6 GB/s**（find_tweet，单核；用"Arm V 系列服务器核"，不暗示鲲鹏=V2）[来源: Lemire 博客] ✅
- SVE 向量长度：**128-2048 bit，128-bit 步进**，VLA（架构规范，无需"约"）[来源: Stephens et al., IEEE Micro 37(2) 2017, arXiv:1803.06185] ✅
- SVE2 SVMATCH：Neoverse-N2 2 cycle/指令，单条指令匹配至多 16 token；Sonic JSON 吞吐提升**最高约 24%**（范围 -0.3%~24.3%，⚠️ 微基准、高度依赖语料）[来源: Arm blog, Yibo Cai 2024-09] ✅
- ⚠️ 不存在 SVE 版 JSON 解析器公开 GB/s 数字（simdjson issue #2683 开放）[来源: 新写，承接 P3-5 校验] ✅
- iceoryx2：常数级 sub-µs、与 payload 无关（⚠️ 数量级；勿暗示快于 raw shmem）[来源: iceoryx2 Discussion #435] ⚠️
- NEON 无 gather/scatter（仅 LD1-LD4 连续/交错）；非连续 AoS 需 vzip/vuzp/vtrn 转置 [来源: ARM ARM] ✅ ISA 事实
- AoS→SoA 转置是 canonical SIMD repack；gather/scatter 远劣于 unit-stride（⚠️ 定性，无精确乘数）[来源: ryg blog; Intel BKM] ✅

========== PAGE 19: P3-6 融合点：同一组单元、同一份布局 — 🎯 prescription climax（主线第三次点题） ==========

## 核心观点（一句话）

**🎯 prescription climax——主线第三次点题/落地。** 传统架构里，comm 的"数据搬运/序列化"与 compute 的"向量化/矩阵化"是**两套单元、两次拷贝**；融合架构是**同一组单元（现役 NEON）、同一份布局、无边界拷贝；SVE/SME 是同一融合方向的下一站**（去 SME 硬依赖）。**通信计算融合，就是我们在 Agent 时代的主场。**

## 详细要点

### 要点1：传统架构痛点——两套单元、两次拷贝

传统架构里，通信与计算被当成两个独立问题分别处理 [来源: 新写，承接 P3 spec §5]：
- **通信侧**：一套机制——HTTP 栈、JSON 解析、拷贝传递、序列化/反序列化。
- **计算侧**：另一套机制——向量化算子、矩阵引擎、SoA 布局。
- **边界**：两者之间必然有**拷贝/序列化/布局转换的边界开销**（AoS↔SoA 转置、pack/unpack、memcpy）。这部分开销在传统架构里无法消除——因为通信栈和计算栈是两套独立的单元、两套独立的内存布局。

### 要点2：融合架构优势——同一组单元、同一份布局（去 SME 硬依赖）

在 ARM NEON/SVE/SME 上，序列化/pack/gather/转置**本身就是 SIMD/矩阵操作** [来源: 新写，承接 P3 spec §5 + 平台文档 §4]：
- **同一组单元**（现役 NEON）：comm 的"数据搬运/序列化"和 compute 的"向量化/矩阵化"用**同一组执行单元**——序列化是 128-bit SIMD 操作，向量化 GEMM 也是 128-bit SIMD 操作，不再是两套单元。
- **同一份布局**：comm 原语与算子共享同一份内存布局（SoA + 连续 ld1/st1），消除两者之间的 AoS↔SoA 转置拷贝。
- **无边界拷贝**：因为单元相同、布局相同，通信和计算之间不再有"两套机制、两次拷贝"的边界开销。
- **SVE/SME 是同一方向的下一站**（去 SME 硬依赖）：SVE 的 predicate+gather/scatter（更宽向量、非连续原语）、SME 的 ZA tile+outer-product（矩阵引擎）是融合的**增量增强**，不是另起炉灶。融合论点**不依赖 SME 才能成立**——现役 NEON 上序列化/pack 就是 SIMD 操作，融合已经在今天发生。

### 要点3：对比图（文字描述）

**左：传统架构**——两个框。上框"通信栈"（HTTP/JSON/拷贝），下框"计算栈"（向量化算子/矩阵引擎）。两框中间一个箭头标注"边界拷贝/序列化/布局转换"——这部分开销在传统架构里是结构性存在、无法消除。

========== PAGE 21: 总结 + 讨论问题（收束） ==========

## 核心观点（一句话）

**thesis 回扣 + 主场收束**：Agent loop 本质是 compute↔comm 交织（P1）→ 从单到多 Agent，通信成主导（P2）→ 在 ARM 平台（NEON 现役 + SVE/SME 前瞻）把 comm 与 compute 融合 = 主场（P3）。**通信计算融合，就是我们在 Agent 时代的主场——现役 NEON 今天就能做，SVE/SME 解锁后是同一融合方向的下一站。**

## 详细要点

### 要点1：thesis 回扣——三部分一条主线

整场用一条主线串起来 [来源: 新写，承接统一主线 spine]：
- **P1（基础/problem setup）**：Agent loop 本质是一段不断在"计算"与"通信"之间来回切换的长循环（🎯 第一次点题）。
- **P2（洞察/diagnosis）**：从单 Agent 到多 Agent，通信占比随规模上升、逐渐压过计算成为主导成本（🎯 第二次点题/handoff：通信变贵了，谁来付）。
- **P3（融合/prescription）**：在我们 ARM 平台（NEON 现役 + SVE/SME 前瞻）把 comm 的"数据搬运/序列化"与 compute 的"向量化/矩阵化"融合进同一组执行单元、同一份布局 = 主场（🎯 第三次点题/climax：同一组单元、同一份布局）。

### 要点2：核心收束——通信计算融合就是主场

**通信计算融合，就是我们在 Agent 时代的主场** [来源: 新写，承接统一主线 spine] ✅：
- **现役 NEON 今天就能做**：向量化序列化（simdjson 式 GB/s）、pack/转置（vzip/vuzp/vtrn）、基础 SIMD 算子、小-GEMM kernel（LibXSMM/LibShaLom 风格）、零拷贝 IPC（iceoryx2）——鲲鹏 920 立即可做。
- **SVE/SME 解锁后是同一融合方向的下一站**：SVE 的 predicate+gather/scatter、SME 的 ZA tile+outer-product 矩阵引擎，是增量增强（鲲鹏 930+/950+ 解锁，公开未确认，路线图）。
- **团队能力高度匹配**：GEMM/Attention、MPI/UCX/SDMA、性能方法论三类能力全部映射到三层机会。
