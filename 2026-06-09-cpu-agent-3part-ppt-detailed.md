========== PAGE 1: 开场：主线一句话 + 议程 ==========

## 核心观点（一句话）

**Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 从单 Agent 的"工具调用往返"，到多 Agent 的"A2A 协作"，**通信占比随规模上升、逐渐压过计算成为主导成本**。而我们的 ARM 平台——**现役 NEON 已能融合，SVE/SVE2/SME（鲲鹏 930+/950+ 解锁）是同一融合方向的下一站**——它不只是算子加速器，更是能把"通信的数据搬运/序列化"与"计算的向量化/矩阵化"放进**同一组执行单元**的融合点。**通信计算融合**，就是我们在 Agent 时代的主场。

## 详细要点

### 要点1：Agent 不是"更大的模型"，而是"带工具的长循环"

传统 LLM 调用是一次 request、一次 response。Agent 是把这个调用放进一个可能长达几十步、上百步的循环里。权威数据：SWE-bench Verified 的标准预算上限是每实例 **最多 250 步、最多 3 美元** [来源: Can Software Engineering Agents Self-Evolve on the Fly? arXiv 2511.13646, 引用 SWE-bench Verified 官方配置]；SWE-bench Pro (Scale AI) 给每个 agent 任务 **最多 50 次工具调用**，作者称该额度 "相当标准" [来源: SWE-Bench Pro, Scale AI, OpenReview 9R2iUHhVfr]。这意味着：一个真实的 coding agent 任务，要发生几十到几百次 "LLM 推理 + 工具调用" 的往返。

### 要点2：在这个循环里，工具执行（不是 LLM）吃掉了一半以上的时间

在 LLM-工具的串行循环中，工具执行（编译、测试、文件读写、web 抓取）**占总请求时间的 35%-61%**，按领域平均：编码任务 60%、深度研究 50%、科研任务 36% [来源: PASTE: Act While Thinking, arXiv 2603.18897, 2026, §2.2.1，基于 SWE-bench / DeepResearchBench / ScholarQA，Gemini-2.5/GPT-5.2/Qwen 实测]。这意味着：即使在 Agent 里，LLM 推理也不是唯一瓶颈，工具执行这个 "非 LLM" 的部分同样主导着端到端延迟。

### 要点3：通信成本随规模上升——从单 Agent 到多 Agent 的关键转折

单 Agent 的瓶颈在 loop 内部效率（工具调用、本地推理、序列化）。但当多个 Agent 协作时，瓶颈变成 **多个 loop 之间的协调开销**——即"通信"。关键数据：多 Agent 比 Agent ~**2-5×** token/成本（经验法则，⚠️ 数量级估算，受任务复杂度、Agent 数量、交互轮数影响）[来源: 承袭旧详细文档 P2-3，综合 MetaGPT / UIUC 研究 arXiv:2505.18286]；Continuum (arXiv:2511.02230) 测得多 Agent 系统调度气泡可达 **~58%**（⚠️ 指任务间隙，非工具调用时间）。**诊断论断：从单 Agent 到多 Agent，通信占比随规模上升、逐渐压过计算成为主导成本。**

### 要点4：ARM 平台的融合机会——不只是算子加速器

我们的 ARM 平台——**现役 NEON（鲲鹏 920 已有）+ SVE/SME（鲲鹏 930+/950+ 路线图）**——它不只是传统意义上的算子加速器，更是能把"通信的数据搬运/序列化"与"计算的向量化/矩阵化"放进**同一组执行单元**的融合点。**NEON 现在就能做基础融合（向量化序列化/pack/基础 SIMD）**；SVE 的 predicate+gather/scatter 与 SME 的矩阵引擎是"下一站"。**通信计算融合，就是我们在 Agent 时代的主场。**

### 要点5：三部分议程——从 problem setup 到 diagnosis 到 prescription

- **第一部分（基础）**：Agent 执行流程 = 计算↔通信的反复交织（8min）—— 让听众把 agent loop 看成"计算↔通信的反复交织"
- **第二部分（洞察）**：从单 Agent 到多 Agent，通信随规模上升、渐成主导（10min）—— 现状盘点 + 数据论证
- **第三部分（融合）**：ARM 平台（NEON 现役 + SVE/SME 前瞻）的通信计算融合 = 主场机会（9min）—— 武器、论点、算子侧、通信侧、路线图

## 关键数据速览

- SWE-bench Verified 单实例上限 **250 步 / 3 美元** [来源: arXiv 2511.13646] ✅
- 工具执行占 agent 总请求时间 **35%-61%**（编码任务高达 60%）[来源: PASTE, arXiv 2603.18897] ✅
- 多 Agent 比 Agent ~**2-5×** token/成本（⚠️ 数量级估算，受任务/AI数量/轮数影响）[来源: 承袭旧详细文档 P2-3]
- Continuum 调度气泡可达 **~58%**（⚠️ 指任务间隙）[来源: Continuum, arXiv:2511.02230]

## 演讲备注 / 过渡语

> "各位好。在开始之前，先把边界划清楚：今天我们不聊怎么训模型、不聊怎么优化单个 token 的推理。我们聊的是——当 LLM 从一个'被问一次答一次'的东西，变成一个会自己写代码、跑测试、查资料的'循环'之后，整个系统的成本结构发生了什么变化。
> 你可能会惊讶：在一个 coding agent 的请求里，真正花在'跑工具'——编译、测试、读写文件——上的时间，占了 35% 到 61%。在编码任务里，这个比例高达 60%。换句话说，一大半的延迟，根本不在 LLM 身上，而在我们系统人最熟悉的那些地方——进程、IPC、序列化、并发。这就是我们今天的主场。
> 更关键的是：从单 Agent 到多 Agent，通信成本会随规模上升，逐渐压过计算成为主导成本。而在我们的 ARM 平台上——现役 NEON 已经能做基础融合，SVE/SME 是下一站——通信计算融合，就是我们在 Agent 时代的主场。
> 今天分三部分：第一部分看 Agent 执行流程的本质；第二部分看从单到多 Agent 的成本结构怎么变；第三部分看 ARM 平台的融合机会。让我们从第一部分开始。"

========== PAGE 2: P1-1 章节封面 + 划边界 ==========

## 核心观点（一句话）

**Agent 运行时 = loop 里被 LLM 光环掩盖的系统层。** 我们不碰训练/推理内核，聚焦的是：感知→推理→规划→工具调用→记忆更新 这个长循环里的 CPU 调度、内存访问、工具通信、资源瓶颈——这正是我们的主场。

## 详细要点

### 要点1：一句话定义"Agent 运行时"

Agent 运行时，不是 LLM 本身的训练/推理，而是 **当 LLM 被放进一个循环里、反复调用工具、反复进出进程边界时，整个端到端的延迟和成本结构**。我们的主场，是这个循环里那些被 LLM 光环掩盖的系统层：CPU 调度、内存访问、工具通信、资源瓶颈。

### 要点2：边界划定——我们不碰什么

我们不讨论：pre-training / fine-tuning、GPU 利用率优化、FlashAttention / PagedAttention 这种单次推理内核优化（虽然它们很重要：FlashAttention 把 HBM 读写从 O(N²) 降到 O(N)，带来约 **2-4×** 加速 [来源: FlashAttention, NeurIPS 2022, Dao et al., arXiv 2205.14135] ✅；PagedAttention 让 vLLM 吞吐提升 **2-4×** [来源: Efficient Memory Management for LLM Serving with PagedAttention, SOSP 2023, Kwon et al., arXiv 2309.06180] ✅）。我们关心的是：Agent loop 里那些"非 LLM"的部分。

### 要点3：边界划定——我们聚焦什么

Agent = 感知→推理→规划→工具调用→记忆更新的长循环，**不是单次 API 调用**。我们的主场，是这个循环里的：
- **CPU 调度**：进程启动、prefork、资源池化
- **内存访问**：向量检索（gather、距离计算）、小批量计算
- **工具通信**：IPC（HTTP/UDS/共享内存）、序列化（JSON/Protobuf/FlatBuffers）
- **资源瓶颈**：锁竞争、负载不均、fail-slow

### 要点4：本章目标——结束时听众把 agent loop 看成什么

结束时，听众应该把 Agent loop 看成 **"计算↔通信的反复交织"**：
- 每一步是不同计算形态（文本处理、LLM 推理、规划、工具调用、向量检索）
- 本地计算 ↔ 跨边界通信（API 调用/IPC/工具执行）的乒乓
- 混合/突发/长尾/依赖复杂——新 workload 的本质

### 要点5：主场定位——为什么这是系统软件的主场

当循环拉长、工具调用变密集，系统层的东西开始决定胜负：每次启动一个工具子进程的开销、每次跨进程传数据的 IPC 延迟、每次序列化上下文的成本、每次锁竞争和共识延迟。这些恰恰是经典系统工程的领域——**GEMM/Attention + MPI/UCX/SDMA**，我们的能力直接映射。

## 关键数据速览

- FlashAttention **2-4×** 加速 [来源: arXiv 2205.14135] ✅
- PagedAttention 吞吐 **2-4×** 提升 [来源: arXiv 2309.06180] ✅
- 工具执行占 agent 总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅

## 演讲备注 / 过渡语

> "现在我们把边界划清楚：今天我们不聊怎么训模型、不聊怎么优化 GPU 上的单个 token 推理。我们聊的是——当 LLM 从一个'被问一次答一次'的东西，变成一个会自己写代码、跑测试、查资料的'循环'之后，整个系统的成本结构发生了什么变化。
> 这个循环里有五个环节：感知、推理、规划、工具调用、记忆更新。我们的主场，是这个循环里那些被 LLM 光环掩盖的系统层——进程、IPC、序列化、并发。这些恰恰是经典系统工程的领域，也是我们 GEMM/Attention + MPI/UCX/SDMA 能力直接映射的地方。
> 下一页，我们把这个循环拆开看：它到底是什么样的一个 loop？"

========== PAGE 3: P1-2 Agent 执行流程 = 一个循环 ==========

## 核心观点（一句话）

**Agent 不是"一次 API"，而是长循环。** 感知→推理→规划→工具调用→记忆更新，这个 loop 会重复几十到几百次（SWE-bench 上限 250 步/$3），每一步触发不同的计算形态——混合/突发/长尾/依赖复杂，这是新 workload 的本质。

## 详细要点

### 要点1：标准循环图——感知→推理→规划→工具调用→记忆更新

Agent 的标准工作流来自 ReAct 范式：每一轮产生 **一个 Thought（推理）、一个 Action（工具调用）、一个 Observation（工具返回）** [来源: ReAct: Synergizing Reasoning and Acting in LLMs, arXiv 2210.03629]。后续 ASE 2025 的轨迹研究系统统计了多个 SOTA agent 的 TAR（Thought-Action-Result）步数分布 [来源: A Study of Thought-Action-Result Trajectories, ASE 2025]。完整循环：**感知（输入处理）→ 推理（LLM 调用）→ 规划（下一步决策）→ 工具调用（执行）→ 记忆更新（上下文累积）→ 回到感知**。

### 要点2：不是"一次 API"而是长循环——SWE-bench 250 步/$3 上限

传统 LLM 调用是一次 request、一次 response。Agent 是把这个调用放进一个可能长达几十步、上百步的循环里。权威数据：SWE-bench Verified 的标准预算上限是每实例 **最多 250 步、最多 3 美元** [来源: Can Software Engineering Agents Self-Evolve on the Fly? arXiv 2511.13646, 引用 SWE-bench Verified 官方配置] ✅；SWE-bench Pro (Scale AI) 给每个 agent 任务 **最多 50 次工具调用**，作者称该额度 "相当标准" [来源: SWE-Bench Pro, Scale AI, OpenReview 9R2iUHhVfr]。也就是说，一个真实的 coding agent 任务，要发生几十到几百次 "LLM 推理 + 工具调用" 的往返。

### 要点3：每一步触发不同的计算形态——循环的异构性

这个循环不是重复同样的计算，每一步都是不同的形态：
- **感知**：文本处理（tokenization、模板渲染）→ CPU 密集、内存带宽敏感
- **推理**：LLM API 调用（远程、高延迟）→ 本地 CPU 空闲窗口
- **规划**：轻量推理/规则匹配 → 计算轻、逻辑复杂
- **工具调用**：重计算（代码分析、profiler）+ IO（文件读写、网络请求）→ CPU/IO 混合
- **记忆更新**：向量检索（gather、距离计算）+ rerank → 内存随机访问、小批量计算

### 要点4：循环特点——混合/突发/长尾/依赖复杂

这是 Agent 区别于传统推理服务的关键特征：
- **混合**：同一次执行里密集计算（工具）与高延迟通信（API）交替出现
- **突发**：工具调用时间分布极不均匀（成功 12 步/$1.21 vs 全部 21 步/$2.52，SWE-agent 数据）
- **长尾**：长链路任务的尾部少量步骤占用不成比例的资源
- **依赖复杂**：工具输出依赖、状态共享、fail-slow（单步慢拖累全局）

### 要点5：新 workload 的本质——传统调度假设失效

面向单一计算类型的调度器无法处理这种 compute/comm 混合模式。Agent 的端到端 wall-clock 约 **18.7 分钟/issue**（4B 模型、SWE-TRACE 实测）[来源: SWE-TRACE, arXiv 2604.14820]；Devin **72% 通过的测试耗时 >10 分钟** [来源: Cognition SWE-bench Technical Report]。这证明了：Agent 是一种全新的、混合/突发/长尾的 workload，传统调度假设失效。

## 关键数据速览

- SWE-bench Verified 单实例上限 **250 步 / 3 美元** [来源: arXiv 2511.13646] ✅
- SWE-bench Pro **50 次工具调用**/任务 [来源: Scale AI, OpenReview 9R2iUHhVfr] ✅
- SWE-agent：成功 **12 步/$1.21**，全部（含失败）**21 步/$2.52** [来源: SWE-agent, NeurIPS 2024, arXiv 2405.15793] ✅
- SWE-TRACE：~**18.7 分钟/issue**（4B 模型）[来源: arXiv 2604.14820] ✅
- Devin **72%** 的成功任务耗时 **>10 分钟** [来源: Cognition SWE-bench Technical Report] ✅

## 演讲备注 / 过渡语

> "现在我们把 Agent 的工作流画出来。看起来很复杂，其实就是五个字一个圈：感知、推理、规划、工具调用、记忆更新。但有两个特征决定了它和普通推理服务完全不一样。
> 第一，它不是一次 API 调用，而是长循环。SWE-bench 的标准预算上限是 250 步、3 美元。一个真实的 coding agent 任务，要发生几十到几百次'LLM 推理 + 工具调用'的往返。
> 第二，每一步都是不同的计算形态——文本处理、LLM 推理、规划、代码分析、向量检索。这导致了混合/突发/长尾/依赖复杂的负载特征。传统面向单一计算类型的调度器，根本处理不了这种 compute/comm 混合模式。

========== PAGE 4: P1-3 拆开循环：每步是什么计算形态 ==========

## 核心观点（一句话）

**循环的每一步，是不同的计算形态，对应不同的资源需求。** 感知（CPU 密集）、推理（远程高延迟→本地 CPU 空闲）、规划（计算轻逻辑复杂）、工具调用（CPU/IO 混合）、记忆更新（内存随机访问/小批量计算）——这是异构负载的根源。

## 详细要点

### 要点1：感知——文本处理（tokenization、模板渲染）

**计算形态**：文本处理、tokenization、模板渲染、JSON 序列化/解析
**资源需求**：CPU 密集、内存带宽敏感
**典型延迟**：JSON 解析（simdjson 单核 GB/s 级）[来源: simdjson 官方/GitHub, 2024] ✅；模板渲染（Jinja2 类引擎）通常毫秒级
**优化机会**：SIMD 加速 JSON 解析（simdjson）、模板预编译、零拷贝格式（FlatBuffers）

### 要点2：推理——LLM API 调用（远程、高延迟）

**计算形态**：远程 LLM API 调用（GPT-4o/Claude/DeepSeek 等）
**资源需求**：网络 I/O 密集，本地 CPU 空闲
**典型延迟**：GPT-4o 平均总响应时间约 **570ms**（TTFT≈562ms）[来源: AILatency 持续监控, 2024; Artificial Analysis, 2024] ✅；Claude 3.5 Sonnet 约 **671ms**[来源: AILatency; Artificial Analysis, 2024] ✅；GPT-4o Mini TTFT 约 **200-400ms**（正常负载）[来源: SitePoint, 2025] ✅；DeepSeek 官方 API 平均响应约 **23.64s/请求**（V4-Pro TTFT 最高可达 **128.46s**）[来源: Medium OpenAI vs DeepSeek 对比, 2025; DeepInfra, 2025] ✅
**关键现象**：API 等待期间本地 CPU 利用率 **~0%**（⚠️ 数量级估算，取决于本地是否有其他任务）

### 要点3：规划——轻量推理/规则匹配

**计算形态**：轻量推理、规则匹配、决策树、状态机
**资源需求**：计算轻、逻辑复杂、CPU 缓存敏感
**典型延迟**：通常微秒到毫秒级（取决于规则复杂度）
**优化机会**：规则引擎 JIT 编译、决策缓存、向量化规则匹配

### 要点4：工具调用——重计算（代码分析、profiler）+ IO（文件读写、网络请求）

**计算形态**：重计算（代码分析、profiler、静态分析、Linter）+ IO（文件读写、网络请求、git 操作）
**资源需求**：CPU/IO 混合、内存带宽敏感、可能触发 fork/exec
**典型延迟**：编译/测试（秒级到分钟级）、文件读写（毫秒级）、网络请求（毫秒到秒级）
**关键数据**：工具执行占总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅；fork 1GB 进程平均 **6.5ms**，50GB 进程平均 **253.9ms**[来源: On-demand-fork: A Microsecond Fork, EuroSys 2021, Purdue University] ✅

### 要点5：记忆更新——向量检索（gather、距离计算）+ rerank

**计算形态**：向量检索（embedding、向量数据库 gather、距离计算）、rerank（cross-encoder）
**资源需求**：内存随机访问密集、小批量计算、内存带宽敏感
**典型延迟**：all-MiniLM-L6-v2 约 **4.2ms/句**（~14,200 句/分钟）[来源: AI Tinkerers Lausanne, 2023; SBERT 官方文档, 2024] ✅；cross-encoder reranker 单文档 **5-50ms**，100 候选可达 **500ms-5s**[来源: OneUptime Cross-Encoder Reranking, 2026] ⚠️ 数量级估算
**优化机会**：CPU 量化（ONNX INT8 **3.08×** 加速 [来源: SBERT Efficiency 文档, 2025] ✅）、SIMD 向量化、SVE gather/scatter、SME 矩阵引擎

========== PAGE 5: P1-4 loop = 计算↔通信的反复交织 —— 🎯 主线第一次点题 ==========

## 核心观点（一句话）

**🎯 Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 把"感知→推理→规划→工具调用→记忆"重画成"本地计算 ↔ 跨边界通信"的乒乓：API 等待期间本地 CPU 利用率~0%（串行流水线的灰色空闲）——这是优化的核心靶点。

## 详细要点

### 要点1：重画循环——从五步到乒乓

把 Agent 的标准循环（感知→推理→规划→工具调用→记忆更新）重画成 **"本地计算 ↔ 跨边界通信"** 的乒乓：
- **本地计算**：感知（文本处理）、规划（规则匹配）、记忆更新（向量检索）、工具调用的本地预处理/后处理
- **跨边界通信**：推理（LLM API 调用）、工具调用（远程执行、IPC、文件/网络 IO）

### 要点2：跨边界通信 = API 调用/IPC/工具执行

**跨边界通信**包括：
- **API 调用**：远程 LLM API（GPT-4o/Claude/DeepSeek）
- **IPC**：进程间通信（HTTP localhost、Unix Domain Socket、共享内存）
- **工具执行**：编译、测试、文件读写、网络请求

这些通信的典型延迟：API 调用 **500ms-5s+**（⚠️ 数量级估算，取决于提供商/负载/输出长度）；HTTP localhost **0.1-1ms**（100-1000µs）[来源: Medium — Beyond HTTP: Unix Domain Sockets, 2023] ✅；Unix Domain Socket 约 **130µs**，TCP loopback 约 **334µs**[来源: NodeVibe, 2024] ✅；共享内存 P99 约 **850ns**[来源: HowTech, 2023] ✅。

### 要点3：关键现象——API 等待期间本地 CPU 利用率~0%

在一次 LLM API 调用期间（**500ms-5s+**），Agent 主机的 CPU 占用接近 **~0%**（⚠️ 数量级估算，取决于本地是否有其他任务）——这是被同步阻塞白白浪费的算力。主流大模型 API 延迟：GPT-4o 约 **570ms**（TTFT≈562ms）[来源: AILatency 持续监控, 2024] ✅；Claude 3.5 Sonnet 约 **671ms**[来源: AILatency; Artificial Analysis, 2024] ✅；GPT-4o Mini TTFT 约 **200-400ms**[来源: SitePoint, 2025] ✅；DeepSeek 官方 API 平均响应约 **23.64s/请求**[来源: Medium OpenAI vs DeepSeek 对比, 2025] ✅。

### 要点4：串行流水线的灰色空闲——优化的核心靶点

当前 Agent 框架普遍采用同步串行循环：等待 API → 取结果 → 做后处理 → 组装下个 Prompt → 等待 API。这种模式下，本地 CPU 在 API 等待期间基本空闲——**灰色空闲区域就是优化的核心靶点**。反向验证：PASTE 通过推测式提前执行工具（在 LLM 思考时并行执行下一个工具），将平均任务完成时间降低 **48.5%**，工具执行吞吐提升 **1.8×**，工具等待时间（stall）减少 **67%**[来源: PASTE, arXiv 2603.18897, §7.2-7.3, 2026] ✅。

### 要点5：🎯 主线第一次点题——Agent loop 本质是 compute↔comm 交织

**Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 从单 Agent 的"工具调用往返"（本地计算 ↔ API 调用），到多 Agent 的"A2A 协作"（多个 loop 之间的消息传递），这个 loop 的本质就是 **compute ↔ comm 的反复交织**。通信占比随规模上升、逐渐压过计算成为主导成本——这是 P2 要论证的核心命题。

========== PAGE 6: P1-5 为什么是"新"workload ==========

## 核心观点（一句话）

**Agent 是混合/突发/长尾/依赖复杂的新 workload。** 传统面向单一计算类型的调度器无法处理 compute/comm 混合模式；SWE-agent 成功 12 步/$1.21 vs 全部 21 步/$2.52 证明突发性——任务时长分布极不均匀，fail-slow 现象显著。

## 详细要点

### 要点1：混合——同一次执行里密集计算与高延迟通信交替出现

同一次 Agent 执行里，密集计算（工具调用：代码分析、profiler、编译、测试）与高延迟通信（API 调用、网络请求、文件 IO）剧烈交替。这是 Agent 区别于传统推理服务的关键：传统推理服务是单一计算形态（矩阵乘），Agent 是 **compute + comm 混合**。工具执行占总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅。

### 要点2：突发——工具调用时间分布极不均匀

**SWE-agent 实测数据**：成功解决的实例中位数 **12 步/$1.21**，全部实例（含失败）均值 **21 步/$2.52**——失败实例消耗的步骤/成本约为成功的近 **2 倍**（fail-slow）[来源: SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering, NeurIPS 2024, arXiv 2405.15793] ✅。这证明：Agent 任务时长分布极不均匀，成功的任务可能 12 步完成，失败的任务可能拖到 21 步甚至更多——**突发性是调度噩梦**。

### 要点3：长尾——长链路任务的尾部少量步骤占用不成比例的资源

长链路 Agent 任务（如复杂 bug 修复、大规模代码重构）的尾部少量步骤占用不成比例的资源。Devin **72%** 通过的测试耗时 **>10 分钟**[来源: Cognition SWE-bench Technical Report] ✅；SWE-TRACE 测得 coding agent 约 **18.7 分钟/issue**（4B 模型）[来源: SWE-TRACE, arXiv 2604.14820, 2024-2026] ✅。长尾效应导致：少数慢任务拖累整体吞吐，尾部延迟主导用户体验。

### 要点4：依赖复杂——工具输出依赖、状态共享、fail-slow

Agent 任务有复杂的依赖关系：
- **工具输出依赖**：下一步的输入依赖上一步工具的输出（编辑-验证循环：55% 的文件编辑后立即跟随终端调用 [来源: PASTE, arXiv 2603.18897, §2.3.1] ✅）
- **状态共享**：多个 Agent 或工具之间共享上下文、记忆、文件系统
- **fail-slow**：单步慢拖累全局（21 步任务中任意一步超时，整个任务延迟飙升）

这种依赖复杂度导致：**推测执行的覆盖有限**（只能预测下一步，不能跳过依赖）、**负载均衡困难**（无法提前知道任务会跑多久）。

### 要点5：传统调度假设失效——面向单一计算类型的调度器无法处理

传统调度器假设：
- **单一计算类型**：所有任务都是同一种计算（如矩阵乘）
- **可预测时长**：每个任务的执行时间相对稳定
- **独立任务**：任务之间依赖弱，可随意重排序

Agent workload 打破所有这些假设：
- **混合计算类型**：compute + comm 混合，每一步资源需求不同
- **不可预测时长**：12 步 vs 21 步，分布极不均匀（SWE-agent 数据）
- **强依赖**：工具输出依赖、状态共享，fail-slow 显著

**结论**：传统面向单一计算类型的调度器（如 HPC 作业调度、传统推理服务调度）无法处理 Agent 的 compute/comm 混合模式。**这为 P2 的多 Agent 协同洞察铺路**——当多个 Agent 协作时，调度问题会更复杂。

## 关键数据速览

- SWE-agent：成功 **12 步/$1.21**，全部（含失败）**21 步/$2.52** [来源: SWE-agent, NeurIPS 2024, arXiv 2405.15793] ✅
- 工具执行占 agent 总请求时间 **35%-61%**（编码任务 60%）[来源: PASTE, arXiv 2603.18897] ✅
- 55% 的文件编辑后立即跟随终端调用（Edit-Verify 模式）[来源: PASTE, arXiv 2603.18897, §2.3.1] ✅
- Devin **72%** 的成功任务耗时 **>10 分钟**[来源: Cognition SWE-bench Technical Report] ✅
- SWE-TRACE：~**18.7 分钟/issue**（4B 模型）[来源: arXiv 2604.14820] ✅
