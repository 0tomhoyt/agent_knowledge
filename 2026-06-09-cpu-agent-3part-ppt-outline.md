## 第一部分：基础 —— Agent执行流程基础（5页, 8min, problem setup）

---

### 【第1页】章节封面 + 划边界

- 一句话定义"Agent 运行时"：不碰训练/推理内核，聚焦 loop 里被 LLM 光环掩盖的系统层
- 边界划定：Agent = 感知→推理→规划→工具调用→记忆更新的长循环，不是单次 API 调用
- 主场定位：这个循环里的 CPU 调度、内存访问、工具通信、资源瓶颈——正是我们的主场
- 本章目标：结束时听众把 agent loop 看成"计算↔通信的反复交织"

> 来源: 新写

---

### 【第2页】Agent 执行流程 = 一个循环

- 标准循环图：感知 → 推理 → 规划 → 工具调用 → 记忆更新 → 回到感知
- 不是"一次 API"而是**长循环**：SWE-bench 任务典型 250 步、成本上限 $3
- 每一步触发不同计算形态：文本处理、LLM 推理、规划、工具调用（重计算·IO）、向量检索·rerank
- 循环特点：混合/突发/长尾/依赖复杂——新 workload 的本质

> 来源: 承袭（SWE-bench 250步/$3 上限）

---

### 【第3页】拆开循环：每步是什么计算形态

- **感知**：文本处理（tokenization、模板渲染）→ CPU 密集、内存带宽敏感
- **推理**：LLM API 调用（远程、高延迟 500ms-5s）→ 本地 CPU 空闲窗口
- **规划**：轻量推理/规则匹配 → 计算轻、逻辑复杂
- **工具调用**：重计算（代码分析、profiler）+ IO（文件读写、网络请求）→ CPU/IO 混合
- **记忆更新**：向量检索（gather、距离计算）+ rerank → 内存随机访问、小批量计算

> 来源: 承袭

---

### 【第4页】loop = 计算↔通信的反复交织

- **重画循环**：把"感知→推理→规划→工具调用→记忆"重画成"本地计算 ↔ 跨边界通信"的乒乓
- **跨边界通信** = API 调用/IPC/工具执行，等待时间 500ms-5s
- **关键现象**：API 等待期间本地 CPU 利用率 ~0%（串行流水线的灰色空闲）
- **第一次点题**：Agent 运行时 = 一段不断在"计算"与"通信"之间来回切换的长循环

> 主线触点: 🎯 主线第一次点题 —— Agent loop 本质是 compute↔comm 交织
> 来源: 承袭（标注数量级：API 500ms-5s, CPU~0%）

---

### 【第5页】为什么是"新"workload

- **混合**：同一次执行里密集计算（工具）与高延迟通信（API）交替出现
- **突发**：工具调用时间分布极不均匀（成功 12 步/$1.21 vs 全部 21 步/$2.52，SWE-agent 数据）
- **长尾**：长链路任务的尾部少量步骤占用不成比例的资源
- **依赖复杂**：工具输出依赖、状态共享、fail-slow（单步慢拖累全局）
- **传统调度假设失效**：面向单一计算类型的调度器无法处理 compute/comm 混合模式

> 来源: 承袭（SWE-agent 数据）；为 P2 diagnosis 铺路

## 第二部分：洞察 —— Agent-to-Agent 洞察报告（7页, 10min, diagnosis）

---

### 【第6页】章节封面 + 诊断问题

- **诊断问题**："从 1 个 agent 到 N 个 agent，成本结构怎么变？"
- **引出关键变量**：单 Agent 瓶颈在 loop 内部效率，多 Agent 瓶颈在**多个 loop 之间的协调开销**——即"通信"
- **诊断论断预告**：从单 Agent 到多 Agent，**通信占比随规模上升、逐渐压过计算成为主导成本**
- 本章结构：现状盘点（共同认知）→ 关键数据 → 我们的洞察（通信怎么流、被什么吃）→ 诊断合成 → 桥到 P3

> 来源: 新写

---

### 【第7页】现状盘点：A2A 正在发生什么

- **主流多 Agent 框架**：AutoGen（2-4 Agent 对话）、MetaGPT（5-7 角色软件公司）、CrewAI、LangGraph——各框架通信机制"各自为政"（HTTP/文件系统/无集合通信）
- **Google A2A 协议（2025）**：标准化 Agent-to-Agent 通信协议的早期信号
- **学术 multi-agent 基准**：多 Agent 协作效率评估的兴起
- **一张"地图"**：框架生态 + 协议趋势 + 学术基准 → 建立 A2A 正在发生的共同认知

> 来源: 🆕 需研究（A2A 框架、协议、基准）

---

### 【第8页】关键数据：多 Agent 比单 Agent 贵多少

- **Token/成本增量**：多 Agent 比 Agent ~**2-5×**（经验法则，⚠️ 非精确测量，受任务复杂度、Agent 数量、交互轮数影响）
- **调度气泡**：Continuum (arXiv:2511.02230) 测得多 Agent 系统调度气泡可达 **~58%**（⚠️ 指任务间隙，非工具调用时间）
- **最慢 Agent 决定端到端延迟**：并行 Agent 中最慢者决定整体延迟（木桶效应）
- **关键洞察**：通信与协调开销（序列化、同步、负载不均）是多 Agent 成本上升的主因

> 来源: 承袭（⚠️ 需重标口径：2-5× 为经验法则，58% 为调度气泡）；数据铺路

---

### 【第9页】我们的洞察①：A2A 数据怎么流

- **通信模式**：点对点（1-1）/广播（1-N）/层级编排（树状）/共享黑板（N-1 共享状态）
- **消息序列化开销**：JSON vs Protobuf vs FlatBuffers——JSON 比 Protobuf 慢 **5-10×**（⚠️ 数量级，受消息大小、字段数影响）
- **通信复杂度**：N 个 Agent 全连接 = O(N²) 通道（10 Agent = 45 条；100 Agent = 4950 条）
- **Agent 消息特征**：频繁、小 payload、低延迟敏感——对通信机制要求极高

> 来源: 承袭（通信模式、序列化、复杂度）；通信成本在哪

---

### 【第10页】我们的洞察②：被锁与不均吃掉的延迟

- **锁竞争代价**：高竞争下程序性能下降 **10-100×**（✅ 已确认）
  - 5-10 个 Agent 频繁写：10-30% 时间花在等锁
  - 10+ 个 Agent 高并发写：50%+ 时间等锁，吞吐量骤降
- **分布式共识延迟**：Raft/Paxos 典型延迟 ms 级（10-50ms，10 节点）
- **负载不均衡**：最慢 Agent 拖累整体（编译 Agent 100% CPU vs 产品 Agent 10% CPU → 端到端被编译 Agent 决定）
- **推荐方向**：Agent 系统通常可容忍**最终一致性**，无状态+消息传递模式延迟仅 1-5μs

> 来源: 承袭（锁竞争 10-100× ✅ 已确认、共识延迟、负载不均）；通信成本在哪

---

### 【第11页】诊断合成：通信随规模上升、渐成主导

- **一张图合成**：单 Agent（通信占比小）→ 小 swarm（通信上升）→ 大 swarm（通信占比主导）
- **数据支撑**：2-5× 成本增量、58% 调度气泡、10-100× 锁竞争、ms 级共识延迟——全指向"通信是上升中的主导成本"
- **诊断论断落地**：从单 Agent 到多 Agent，**通信占比随规模上升、逐渐压过计算成为主导成本**
- **为 P3 铺路**：如果通信是主导成本，而我们的 ARM 平台能把通信和计算放进同一组 SIMD 单元……

> 主线触点: 🎯 diagnosis 落地 —— 通信随规模上升、渐成主导
> 来源: 综合（P2-3/4/5 收成一句话论断）

---

### 【第12页】桥到 P3：通信变贵了，谁来付？

- **只抛问题、不回答**："如果通信是上升中的主导成本，而我们 ARM 平台（现役 NEON + 前瞻 SVE/SME）能把通信和计算放进同一组 SIMD 单元……"
- **制造张力**：传统架构——通信的数据搬运/序列化与计算的向量化/矩阵化是**两套单元、两次拷贝**；融合架构——**同一组单元、同一份布局**
- **期待制造**：在 ARM 平台上，"通信计算融合"不只是潜在机会，而是有论证的**主场答案**

> 主线触点: 🎯 主线第二次点题（handoff）——制造对 P3 的期待

## 第三部分：融合 —— ARM 平台通信计算融合（7页, 9min, prescription）

---

### 【第13页】章节封面 + 主场一句话

- **承接 P2-7 问题**："通信变贵了，谁来付？——在我们 ARM 平台上，NEON（现役）+ SVE/SME（前瞻）是通信计算融合的物理基础。"
- **主场一句话**：**现役 NEON 已能做基础融合（向量化序列化/pack/基础 SIMD）；SVE 的 predicate+gather/scatter 与 SME 的矩阵引擎是"下一站"——它们共享同一融合主线**
- **本章叙事弧**：武器（NEON/SVE/SME 是什么）→ 融合论点（三件事）→ 算子侧 → 通信侧 → 融合 punchline → 路线图

> 主线触点: handoff 落地
> 来源: 新写

---

### 【第14页】NEON（现役）/ SVE（前瞻）/ SME（前瞻）——ARM 的三类向量/矩阵武器，为什么对 Agent 特别

- **NEON（现役·鲲鹏 920 现在就有）**：固定 128-bit SIMD（16×8-bit / 8×16-bit / 4×32-bit / 2×64-bit lane），整向量操作、无 per-lane 谓词、无原生 gather/scatter
- **SVE（前瞻·鲲鹏 930+/950+ 解锁）**：向量长度可扩展（128-2048 bit）、**Predicate 寄存器**（逐 lane 谓词）、**原生 gather-load/scatter-store**（向量化非连续访问）
- **SME（前瞻·鲲鹏 930+/950+ 解锁）**：**ZA tile 寄存器**（VL×VL 的 2D 数组）、**Outer-product 引擎**（C += A⊗B，外积之和 = 矩阵乘）
- **为什么对 Agent 特别**：Agent 负载是"变长/不规则/小批量"的——predicate 处理变长、gather/scatter 处理不规则消息访问、矩阵引擎处理小批量 GEMM/embedding

> 来源: ✅ 已研究（见研究文档 docs/superpowers/research/2026-06-09-platform-sme-sve.md §3）；NEON-now + SVE/SME-roadmap 口径

---

### 【第15页】通信计算融合 = 三件事（核心 thesis 页）

- **① 重叠**：通信期间做计算（延迟隐藏）——API 等待 500ms-5s 期间，本地 CPU 利用 SIMD 做数据预处理/后处理/向量检索
- **② 数据搬运即计算**：序列化/pack/转置/gather 本身就是向量化/矩阵化工作——NEON 现役就能做基础 SIMD 加速，SVE/SME 解锁后 predicate/gather-scatter/矩阵引擎是增量增强
- **③ kernel+原语协同**：comm 原语与算子共享 SIMD 单元+内存布局，消除边界拷贝——同一组 NEON/SVE/SME 单元、同一份数据布局
- **论点**：在现役 NEON 上**今天就能做融合**（向量化序列化/pack/基础 SIMD）；SVE 的 predicate+gather/scatter 与 SME 的矩阵引擎是"下一站"——前后是同一融合主线的现在与未来

> 主线触点: 🎯 thesis 展开 —— 通信计算融合的三件事
> 来源: 综合；NEON-now + SVE/SME-roadmap 口径

---

### 【第16页】算子侧：小批量 Agent kernel on ARM SIMD

- **Agent 算子特征**：embedding/rerank/小批量 GEMM/attention 是"瘦高"矩阵（batch 1-8 vs 训练 32-1024）
- **NEON（现役）机会**：128-bit 固定宽度 SIMD 做基础向量化——小批量 GEMM、embedding 聚合、attention 的向量化片段
- **SME（前瞻）增量**：outer-product + ZA tile 天然适配"瘦高"小批量矩阵（区别于训练大 batch）；矩阵引擎让小批量也有高占满度
- **映射团队能力**：GEMM/Attention 优化经验 → Agent-native 小批量 kernel 库（⚠️ 若无 SME 实测数据，标"数量级 + 可自测 micro-benchmark"）

> 来源: 🆕 需研究（SME 小批量 GEMM 数据）；NEON-now（基础向量化）+ SME-roadmap（矩阵引擎增量）

---

### 【第17页】通信侧：SIMD 加速 Comm 数据通路

- **序列化加速**：NEON 现役就能做 simdjson 式 JSON/Protobuf 解析（128-bit SIMD 处理字段）；SVE 的 predicate 处理不规则字段是增量增强
- **共享内存零拷贝**：跨 Agent 消息传递用共享内存+指针传递（vs HTTP 1-5ms → 共享内存 100ns，差距 10,000-50,000×）
- **消息搬运加速**：SVE 原生 gather/scatter 做非连续消息体搬运；SME ZA tile 做 on-the-fly pack/转置
- **映射团队能力**：MPI/UCX/SDMA → Agent-native 通信原语（零拷贝 IPC、向量化序列化）

> 来源: 承袭（simdjson 数据）+ 🆕 SVE 落地；NEON-now（simdjson 式解析）+ SVE/SME-roadmap（gather/scatter/tile 增量）

---

### 【第18页】融合点：同一组单元、同一份布局（punchline）

- **传统架构痛点**：comm 的"数据搬运/序列化"与 compute 的"向量化/矩阵化"是**两套单元、两次拷贝**——边界有冗余开销
- **融合架构优势**：在 NEON/SVE/SME 上，序列化/pack/gather 本身就是 SIMD/矩阵操作——**同一组单元、同一份布局、无边界拷贝**
- **对比图**：传统（边界拷贝/序列化开销）vs 融合（comm 数据搬运 = compute 向量化操作）
- **收束论点**：**通信计算融合，就是我们在 Agent 时代的主场**——现役 NEON 今天就能做，SVE/SME 解锁后是同一方向的下一站

> 主线触点: 🎯 prescription climax，主线第三次/落地 —— 同一组单元、同一份布局
> 来源: 综合；NEON-now + SVE/SME-roadmap 口径（去 SME 硬依赖）

---

### 【第19页】怎么开始：三层机会 + 路线图

- **短期（现在-6 个月，NEON 现役）**：SIMD JSON/Protobuf 算子（simdjson 式）、零拷贝 IPC 原型、基础向量化 pack/转置——**现役鲲鹏 920 立即可做**
- **中期（6-18 个月，SVE/SME 解锁）**：SVE predicate+gather/scatter 版序列化、SME 小批量 Agent kernel 库（embedding/rerank/小批量 GEMM）、Agent-native 通信原语标准化
- **长期（18-36 个月）**：Agent Workload Benchmark + CPU-Agent 协同设计反哺架构（向 ARM 反馈 Agent-native 特性需求）
- **能力映射**：GEMM/Attention → 小批量 kernel；MPI/UCX/SDMA → 通信原语；性能方法论 → Benchmark

> 来源: 承袭 roadmap；NEON-now（短期全部可做）+ SVE/SME-roadmap（中长期）

### 【第20页】开场：主线一句话 + 议程

- **三部分议程**：

  - 第三部分（融合）：ARM 平台（NEON 现役 + SVE/SME 前瞻）的通信计算融合 = 主场机会（9min）
