## 1. 重构动机与统一主线

### 1.1 为什么重构

旧结构是"问题驱动"（瓶颈分析 → 前瞻方向），四部分、27 正文页，三个独立 mini-talk 之间叙事弧偏散。作者希望改为**三部分**：

1. **Agent 执行流程的基础分享介绍**（主打基础）
2. **Agent-to-Agent 洞察报告**
3. **通信计算融合在 ARM CPU、SME、SVE 下的潜在机会**

并在三个关键约束下重构：

- **硬件**：团队主力平台为 **ARM 服务器 CPU**（倚天/鲲鹏/飞腾/Graviton 类），**SME/SVE 是已有或正在跟进的能力**。因此第三部分是"深化、落地我们正在做的事"，而非前瞻论证一个新方向。
- **P2 定位**："洞察报告" = **两者结合**——先快速盘点业界/学术现状（建立共同认知），再落到团队提炼的关键洞察与瓶颈，桥到第三部分。
- **时间**：**均衡分配**，约 P1 8min / P2 10min / P3 9min + 开场 1min + 总结 2min = 30min。

### 1.2 统一主线（全场一句话 spine）

> **Agent 运行时 = 一段不断在「计算」与「通信」之间来回切换的长循环。** 从单 Agent 的"工具调用往返"，到多 Agent 的"A2A 协作"，**通信占比随规模上升、逐渐压过计算成为主导成本**。而我们的 ARM 平台拥有 **SME（矩阵）+ SVE（向量）**——它们不只是算子加速器，更是能把"通信的数据搬运/序列化"与"计算的向量化/矩阵化"放进**同一组执行单元**的融合点。**通信计算融合**，就是我们在 Agent 时代的主场。

三部分构成 **problem setup → diagnosis → prescription** 的诊断弧：

| 部分 | 在主线中的角色 |
|------|----------------|
| **P1 基础** | **problem setup** — 让听众从第 1 页就看到 agent loop 本质是 compute↔comm 交织（非"什么是大模型"那种浅基础） |
| **P2 洞察** | **diagnosis** — 现状盘点 + 数据论证："单→多 Agent，通信占比上升、渐成主导" |
| **P3 融合** | **prescription** — 在 ARM 平台用 SME/SVE 把 comm 与 compute 融合 = 主场机会 |

主线把作者最初对 P3 的措辞"潜在机会"升级为"有论证的主张"——机会不是清单，而是诊断弧的自然结论。

## 3. Part 1 基础 —— Agent 执行流程基础（5 页, 8min, problem setup）

**目标**：结束时听众把 agent loop 看成"计算↔通信的反复交织"，并理解为什么这是新 workload。**不讲"什么是大模型/神经网络"**——受众是系统软件工程师，要的是 agent 运行时视角。

| 页 | 标题 | 要点 | 主线触点 | 来源 |
|----|------|------|----------|------|
| P1-1 | 章节封面 + 划边界 | 一句话定义"Agent 运行时"；我们不碰训练/推理内核，讲 loop 里被 LLM 光环掩盖的系统层 | — | 新写 |
| P1-2 | Agent 执行流程 = 一个循环 | 感知→推理→规划→工具调用→记忆更新 循环图；不是"一次 API"而是几十~几百步长循环 | — | 承袭（SWE-bench 250步/$3 上限） |
| P1-3 | 拆开循环：每步是什么计算形态 | 感知(文本处理)/推理(LLM)/规划/工具调用(重计算·IO)/记忆(向量检索·rerank)→映射到 CPU/内存/通信需求表 | — | 承袭 |
| **P1-4** | **loop = 计算↔通信的反复交织** | 把循环重画成"本地计算 ↔ 跨边界通信(API/IPC/工具)"的乒乓；API 等待 500ms-5s 间隙本地 CPU 利用率~0% | **🎯 主线第一次点题** | 承袭（标注数量级） |
| P1-5 | 为什么是"新"workload | 混合/突发/长尾/依赖复杂/fail-slow（成功 12 步/$1.21 vs 全部 21 步/$2.52）；传统调度假设失效 | 为 P2 diagnosis 铺路 | 承袭（SWE-agent） |

**承袭率**：P1 几乎全是现有已验证内容的**蒸馏**（SWE-bench、SWE-agent、API 间隙 CPU~0%、workload 特征均已过校验），新研究量很低。P1-4 是主线 spine 的第一次显式亮相。

**可选第 6 页**（若 5 页讲 8min 偏松）：一张"本地 CPU 在忙什么"profiling 示意图（串行流水线灰色空闲 vs 并行填满），视觉化优化目标。

## 4. Part 2 洞察 —— Agent-to-Agent 洞察报告（7 页, 10min, diagnosis）

**角色**：主线的 **diagnosis**。按"两者结合"——**先现状盘点（建立共同认知）→ 再我们提炼的洞察 → 桥到 P3**。诊断论断：**"从单 Agent 到多 Agent，通信占比随规模上升、渐成主导。"**

| 页 | 标题 | 要点 | 主线触点 | 来源 |
|----|------|------|----------|------|
| P2-1 | 章节封面 + 诊断问题 | "从 1 个 agent 到 N 个 agent，成本结构怎么变？"——引出"通信"这个新变量 | — | 新写 |
| P2-2 | 现状盘点：A2A 正在发生什么 | 主流多 Agent 框架速览（AutoGen/MetaGPT/CrewAI/LangGraph）+ Google **A2A 协议**(2025) + 学术 multi-agent 基准；一张"地图"建立共同认知 | — | **🆕 需研究** |
| P2-3 | 关键数据：多 Agent 比单 Agent 贵多少 | token/成本 ~2-5×（经验法则，**标注非精确**）；调度气泡可达 ~58%（Continuum, arXiv:2511.02230）；最慢 agent 决定端到端延迟 | 数据铺路 | 承袭（**需重标口径**） |
| P2-4 | 我们的洞察①：A2A 数据怎么流 | 点对点/广播/层级编排/共享黑板；消息序列化开销（JSON vs Protobuf vs FlatBuffers）；随 agent 数通信复杂度 O(N²) vs O(N log N) | 通信成本在哪 | 承袭 |
| P2-5 | 我们的洞察②：被锁与不均吃掉的延迟 | 锁竞争 10-100× 退化（**已确认✅**）；分布式共识延迟（Raft/Paxos ms 级）；负载不均衡（最慢 agent 拖累整体） | 通信成本在哪 | 承袭 |
| **P2-6** | **诊断合成：通信随规模上升、渐成主导** | 一张图：单 agent（通信占比小）→ 小 swarm → 大 swarm（通信占比主导）；把 P2-3/4/5 收成一句话论断 | **🎯 diagnosis 落地** | 综合 |
| **P2-7** | **桥到 P3：通信变贵了，谁来付？** | 只抛问题、不回答："如果通信是上升中的主导成本，而我们 ARM 平台能把通信和计算放进同一组 SME/SVE 单元……" | **🎯 主线第二次点题（handoff）** | 新写 |

**承袭与新研究**：
- **承袭为主**：P2-4（通信模式/序列化）、P2-5（锁/共识/负载）几乎全是现有已验证内容。
- **P2-3 必须重标口径**：仓库校验报告已警告——"2-5×"是 token/成本经验法则、"58%"是调度气泡而非工具调用时间。PPT 按**数量级 + 限定语境**引用，不能当精确测量。
- **🆕 新研究集中一处**：P2-2 的 A2A 现状盘点（框架生态 + A2A 协议 + 学术基准）是现有文档没有的。

**叙事设计**：P2-7 是"只问不答"的 handoff 页——制造对 P3 的期待，把诊断张力推到最高，由 P3 给出 prescription。

## 5. Part 3 融合 —— ARM SME/SVE 通信计算融合（7 页, 9min, prescription）

**角色**：主线的 **prescription**。承接 P2-7 抛出的问题，给出主场答案：**ARM 平台的 SME（矩阵）+ SVE（向量）是"通信计算融合"的物理基础**。叙事弧：**武器（SME/SVE 是什么）→ 融合论点（三件事）→ 算子侧 → 通信侧 → 融合 punchline → 路线图**。

| 页 | 标题 | 要点 | 主线触点 | 来源 |
|----|------|------|----------|------|
| P3-1 | 章节封面 + 主场一句话 | 承接 P2-7，一句话："在我们 ARM 平台上，SME+SVE 是通信计算融合的物理基础。" | handoff 落地 | 新写 |
| P3-2 | SME/SVE 是什么、为什么对 agent 特别 | SVE（向量·predicate·gather/scatter·向量长度可扩展）+ SME（2D tile·outer-product 矩阵引擎）；论点：**可扩展宽度+predicate+gather/scatter** 匹配 agent 的"变长/不规则/小批量"，vs x86 AVX-512 固定宽度 | 武器亮相 | **🆕 需研究**（spec + **平台型号/是否有 SME**） |
| **P3-3** | **通信计算融合 = 三件事**（核心 thesis 页） | ①**重叠**：comm 期间做计算（延迟隐藏）②**数据搬运即计算**：SME/SVE 加速序列化/pack/转置/gather——comm 的数据搬运本身就是向量化/矩阵化工作 ③**kernel+原语协同**：comm 原语与算子共享 SME/SVE 单元+内存布局，消除边界拷贝 | **🎯 thesis 展开** | 综合 |
| P3-4 | 算子侧：小批量 agent kernel on SME | agent 的 embedding/rerank/小批量 GEMM/attention 是"瘦高"矩阵，SME outer-product+tile 适合；vs 训练推理大 batch GEMM。映射团队 **GEMM/Attention** 能力 | compute 侧机会 | **🆕 需研究**（SME 小批量 GEMM 数据） |
| P3-5 | 通信侧：SVE 加速 comm 数据通路 | SVE 版 simdjson 式 JSON/序列化（承袭 simdjson 数据，落到 SVE/NEON）；共享内存零拷贝+SVE gather/scatter 搬运不规则消息；pack/transpose 用 SME tile。映射团队 **MPI/UCX/SDMA** 能力 | comm 侧机会 | 承袭 + **🆕 SVE 落地** |
| **P3-6** | **融合点：同一组单元、同一份布局**（punchline） | 合成 P3-4+P3-5：comm 的"数据搬运"与 compute 的"向量化/矩阵化"在 SME/SVE 上是**同一类操作、共享单元和布局**，所以能融合。对比图：传统（边界有拷贝/序列化开销）vs 融合（无边界） | **🎯 prescription climax，主线第三次/落地** | 综合 |
| P3-7 | 怎么开始：三层机会 + 路线图 | 短期（SVE 序列化/JSON 算子、零拷贝 IPC 原型）→ 中期（SME 小批量 agent kernel 库、comm 原语标准化）→ 长期（Agent workload benchmark + CPU-Agent 协同设计反哺架构） | 可执行收口 | 承袭 roadmap |

**承袭与新研究**：
- **承袭**：P3-5（simdjson 数据）、P3-7（三层时间线、Agent-native 算子/原语/benchmark 前瞻）来自现有已验证内容，现在 ARM 落地。
- **🆕 新研究（P3 是研究重灾区，必须并行 + 对抗式校验）**：
  - **P3-2**：SME/SVE 规格细节 + **关键问题：我们平台型号到底有没有 SME**（倚天/鲲鹏/Graviton 的 SVE 宽度、SME 可用性差异大——若暂无 SME，P3-4 口径从"已具备"调整为"路线图/即将具备"，**必须先查实再定口径**）。
  - **P3-3**：通信计算融合 / overlap 在 ARM 上的学术+工业先例。
  - **P3-4**：SME 小批量 GEMM/embedding 实测或论文数据（区别于训练大 batch）。
  - **P3-5**：SVE 版本序列化/JSON、gather/scatter 用于消息搬运的数据。

**叙事设计**：P3-3 抛"融合三件事"概念 → P3-4/P3-5 从 compute/comm 两侧铺垫 → P3-6 合成 punchline（同一组单元、同一份布局）= 全场高潮 → P3-7 可执行收口。

## 11. 关键风险与开放问题

- **R1（最高）**：平台 SME 可用性未查实 → **P3 整体口径悬空**（不只 P3-4：P3-3 thesis、P3-4 SME kernel、P3-5 SME tile 转置、P3-6 "同一组单元=SME+SVE" punchline、P3-7 SME kernel 库都依赖 SME）。作者硬件答复为"SME/SVE 是已有或**正在跟进**的能力"，故 SME 可能尚未到位。**缓解**：列为实现第一步阻塞项。查实后分两种口径——
  - **若已有 SME**：按原设计（P3-4 = 已具备的算子机会）。
  - **若 SME 尚未到位**：P3 整体**pivot 为"SVE-first 融合 + SME 路线图"**——thesis 强调 SVE 向量/predicate/gather-scatter 已可融合 comm 数据通路，SME 矩阵引擎作为"即将解锁"的算子侧升级写在 roadmap（P3-7）与 P3-4 的"前瞻"口径里。P3-6 punchline 改述为"同一组向量单元、同一份布局"（去 SME 依赖）。
- **R1 决议（实现期）**: 口径 = NEON-now + SVE/SME-roadmap（比 spec 原两分支更靠前：现役鲲鹏 920 连 SVE 都没有，仅 NEON）；依据 = 鲲鹏 920（TaiShan v110, ARMv8.2-A）经多源确认只具备 NEON（固定 128-bit SIMD），无 SVE/SVE2/SME（IEEE Micro 2021 官方芯片论文 + VLDB Journal 2022 + Stony Brook Ookami），且鲲鹏 930 从未正式发布、其 SVE2/SME 状态公开未确认；详见 `docs/superpowers/research/2026-06-09-platform-sme-sve.md`。
- **R2**：SME 小批量 agent kernel 的公开实测数据可能稀缺（学术多关注训练大 batch GEMM）。**缓解**：若无权威实测，标注为数量级估算 + 团队可自测的 micro-benchmark 方案。
- **R3**：A2A 协议（Google, 2025）较新，生态数据可能不稳。**缓解**：作为"趋势信号"而非精确基准引用。
- **R4**：均衡 21 页可能仍偏紧（P3 有 7 页但要承载武器+三件事+两侧+punchline+roadmap）。**缓解**：P3-2/P3-7 可压成"封面信息密度高"的页；必要时启用附录承接细节。
